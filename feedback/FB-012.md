Minor data-quality issue noticed while creating a `compliance-gap` element
(CG-DCR-001) during the foundational run.

The `gapStatus` frontmatter field on `compliance-gap` elements is required,
but its value is not normalised. Across the existing wiki:

    $ grep -rh "^gapStatus:" wiki/processes/*/control-gaps/
    gapStatus: OPEN
    gapStatus: open

Both casings are in use. Whichever a skill happens to type is what gets
written — `write_element.py` and `check_conformance.py` accept either, because
the field has no enum.

Impact is small but real: any code, filter or roll-up that matches `gapStatus`
by string equality ("show me the open gaps") will silently miss half of them,
and the inconsistency looks sloppy in an audit-facing tool.

**Suggested fix.** Give `gapStatus` an enum in `schema/process-schema.json`
(e.g. `OPEN` / `CLOSED` / `ACCEPTED`) the same way `severity`, `impact` and
`controlType` are enumerated, and have `check_conformance.py` enforce it.
Optionally normalise the existing values in one migration pass.
