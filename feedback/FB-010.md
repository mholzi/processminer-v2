Observed while running `comment-review` on PS-DCR-002 (Verify customer
identity).

The SME comment asked the step to record the real verification threshold —
"a fixed set of three security questions, caller must answer at least two
correctly to pass" — instead of the generic "security questions". That is a
valid [I] Incorporate, and the "What happens" block was patched accordingly.

**The gap.** That exact fact — the three-questions / pass-two threshold — is
*also* the substance of **CP-DCR-001**, the control "Customer identity
verification before any card action", whose "Control activity" block still
reads only "verifies identity through knowledge-based security questions". The
step and the control now describe the same real-world check at two different
levels of precision.

`comment-review`'s scope is explicit and narrow: "You edit only that element
... You review the comments on **one** element per run." So the skill is
*correct* not to touch CP-DCR-001 — but it also has **no mechanism to flag**
that the incorporated change implicates a linked element. The summary note can
mention it (this run's summary did, by hand), but nothing structured is
recorded and nothing re-opens or tags CP-DCR-001.

The SME only catches the drift if they happen to run `run-lint` afterward — and
the comment-review flow gives them no nudge to.

**Suggested improvement.** When an [I] incorporation changes a fact, have
`comment-review` check the edited element's relations (here, `PS-DCR-002` →
`CP-DCR-001` via the control's `step` link) and surface "this change may
implicate: CP-DCR-001" — either as a line in the closing summary, a discussion
comment on the implicated element, or a lint-style finding. Improving one
element while a linked element silently goes stale is exactly the
cross-element inconsistency the tool exists to prevent.
