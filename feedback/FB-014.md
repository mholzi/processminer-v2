Found while running `client-journey-specialist` on the Debit Card Replacement
process. Same family as FB-013, and proof that FB-013's fix was too narrow.

Every specialist skill is told to call `show_template.py <type>` and trust its
output: *"That is the full contract — you do **not** read the whole schema
file."* But `show_template.py` prints only a field's **key** — it drops the
schema's `hint` and `suffix`, which are the metadata that says what the field
actually expects.

**It caused twelve wrong values in this run.** `cx-touchpoint` has two scalar
fields whose format is entirely carried by metadata `show_template` does not
print:

- `channel` — schema `hint`: *"The channel this touchpoint occurs on (e.g.
  branch, app, call centre)"* — i.e. a channel **name**.
- `ces` — schema `hint`: *"Customer Effort Score for this touchpoint"*, plus
  `suffix: " / 7"` — i.e. a **number, 1–7**.

`show_template.py cx-touchpoint` printed them as a bare:

    fields:
      channel
      ces

With no hint and no suffix, the session guessed — and guessed wrong on both:
`channel` was written as the `CH-…` element id (not a name), `ces` as
`Low` / `Medium` / `High` (not a 1–7 number). Six touchpoints × two fields =
twelve wrong values. `check_conformance.py` passed them all — it does not
validate free-text field content — so the error was caught only by manually
grepping how existing touchpoints in other processes store the fields.

**Why it matters.** The skill instruction and the tool disagree: the skill
says "do not read the schema file", the tool then withholds the part of the
schema a skill needs to fill a field correctly. FB-013 fixed exactly one
symptom of this — the missing `urlKey` companion — but `hint` and `suffix`,
which every field can carry, are still dropped.

**Suggested fix.** Have `show_template.py` print each field's `hint` (and
`suffix` where set) alongside its key — e.g.

    fields:
      channel   — The channel this touchpoint occurs on (e.g. branch, app, call centre).
      ces       — Customer Effort Score for this touchpoint, if measured.  [value + " / 7"]

The hint already exists in the schema for almost every field; surfacing it is
a few lines in `print_type()`. Consider also giving `check_conformance.py` a
light type check (a field documented as numeric should not hold `High`).
