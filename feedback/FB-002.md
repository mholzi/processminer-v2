During a full front-to-back run I typed "create a new process" into the chat
while another process was open. The assistant correctly refused — the chat
session is scope-locked to one process (`scopePreamble` in
`ProcessDocScreen.tsx`) — but the decline message read:

> "To create a new process, please start a fresh session without the SPP
> scope and use the /new-process skill."

That instruction is written for a Claude Code CLI user. The SME using the web
app has no "fresh session" and no `/new-process` slash command — they have the
**+ New process** button in the top bar, which is the actual supported path
(it calls `createProcess()` → unscoped `new-process`).

The scope-lock behaviour itself is correct; only the guidance is wrong.
Rule 3 of the scope preamble should tell the assistant to point the SME at
the "+ New process" button in the top bar rather than at a CLI skill. As
written it leaks the implementation model and dead-ends a non-technical user.
