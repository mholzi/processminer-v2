The single biggest workflow snag in this F2B run. It is a contradiction
*inside* the skill instructions, not a one-off mistake.

**What happens.** `document-ingest` marks each element's *interpretive* blocks
`provenance: proposed` — "Why it matters" on every process-step, "Impact" on
every exception, "Risk addressed" on every control, "Why it matters" on every
metric. In this run that was at least one `proposed` heading on **all 23**
ingested elements.

`set_approval.py` has a hard gate — `gate_approval()` — that **blocks
`approved`** while any heading is `proposed` (or `web`):

    error: PS-DCR-001 cannot be approved — these headings are not yet
    confirmed by the SME: Why it matters. Walk each one through the SME ...

The foundational-run skill's Step 3, outcome **[Y]**, says only:

> **[Y]** — the SME is satisfied. Run `python3 scripts/wiki/set_approval.py
> <slug> <id> approved "<SME name>"`.

So an agent that follows the [Y] branch literally — challenge the element, SME
says [Y], call `set_approval.py` — **fails on the first element and every
element after it**. The happy path of the skill's core flow does not work as
written.

**Why it's a contradiction, not just a missing line.** The fix *is* in the
skill — but only in the Provenance block at the very bottom: "When the SME
confirms a `proposed` heading in read-back, rewrite the element with that
heading marked `elicited`." Two problems:

1. The **[Y] branch never references it.** "The SME is satisfied → approve"
   reads as a one-liner. Nothing at the decision point tells the agent that
   approval first requires rewriting provenance.
2. The **challenge step (item 2) never walks the proposed headings.** It asks
   for "1–3 sharp questions designed to expose weakness" — it does not
   enumerate the element's `proposed` headings and read each one back. So even
   a faithful run can reach [Y] without having confirmed the headings the gate
   checks.

The result: the two halves of the skill (challenge → [Y] → approve) and the
gate they must satisfy were specified independently and do not meet.

**How it was worked around in this run.** Only because the run inspected
`set_approval.py`'s source first — then every element's `proposed` headings
were flipped to `elicited` *before* calling `set_approval.py`. An agent
trusting the skill text would not know to do that.

**Suggested fix.** Make the [Y] branch explicit: before `set_approval.py`,
read back every `proposed` heading to the SME and rewrite the element with the
confirmed ones marked `elicited` — and have the challenge step (item 2)
require walking each `proposed` heading, so "satisfied" genuinely means
"every heading confirmed". Either that, or `set_approval.py` should report
*which* call site is expected to have flipped them, so the failure is
self-explaining.
