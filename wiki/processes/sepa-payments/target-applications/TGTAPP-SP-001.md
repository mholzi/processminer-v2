## Rationale
Keep the existing Payment Hub but re-platform it to event-driven and ISO 20022-native. Build is too risky given vendor scheme certification; full replacement loses years of routing rule edge cases.

## Tech stack
Java 21 on Kubernetes (EU-Frankfurt + EU-Amsterdam), PostgreSQL for transactional state, Kafka for the event backbone, gRPC inward to the risk engine.

## Risks
- Vendor's modernised release ships H2 2026; a slip slips SCT Inst go-live with it.
- In-flight routing-rule migration is non-trivial — dedicated workstream needed.
- Test environment availability with the vendor is the binding constraint.
