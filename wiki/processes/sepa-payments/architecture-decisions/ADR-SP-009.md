## Context
Event-driven flows imply at-least-once delivery semantics. SCT Inst payments must not double-debit a customer because a Kafka consumer retried after a transient failure. Today there is no consistent idempotency key across components.

## Decision
Use the ISO 20022 EndToEndId as the primary idempotency key across every component. Each component stores `(component, EndToEndId, request-hash) → outcome` for 30 days; a repeat returns the prior outcome unchanged.

## Alternatives considered
- Service-level uuids minted per hop.
- An idempotency token issued by the channel.
- Rely on Kafka exactly-once semantics only.
- A global idempotency service every component calls.

## Consequences
- Storage cost is small per component — 30-day window is tractable.
- EndToEndId is customer-supplied; channels must mint one if absent.
- Replays are safe by design — operational confidence improves.
- Late-arriving duplicates beyond 30 days are unhandled; rare but documented.
- Audit trail naturally keys on EndToEndId, simplifying investigations.
