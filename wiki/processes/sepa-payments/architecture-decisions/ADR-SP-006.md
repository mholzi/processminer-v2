## Context
The Payment Hub holds short-lived transactional state per payment, while the Real-Time Risk Engine needs to query customer activity histories spanning months for behavioural features. One database does not fit both shapes; forcing it costs latency or capacity.

## Decision
PostgreSQL clusters back the payment hub's transactional state. Cassandra backs the risk engine's time-series feature store and the audit trail. Both replicated across two EU regions for active-active.

## Alternatives considered
- PostgreSQL everywhere; partition the activity history aggressively.
- Cassandra everywhere; accept eventual consistency for the hub state.
- Use the vendor's managed datastore offerings.
- Use a single graph database to model both shapes.

## Consequences
- Two database stacks, two operational learning curves.
- Cross-store joins are application-side — design discipline required.
- Backup, restore and disaster-recovery procedures double in surface area.
- Right-tool-for-the-job latency profile.
- Cassandra hiring pool in the EU is smaller than PostgreSQL — plan ahead.
