What was run, in order

  1. new-process — created the "Corporate Account Closure" process (slug corporate-account-closure, abbreviation CAC) via the chat.
  2. document-ingest — uploaded a mock DTP through the upload modal's paste mode; extracted 38 draft elements (8 steps, 5 roles, 7 controls, 5 systems, 4 exceptions, 5 metrics, 4 process-gaps), 2 verification corrections.
  3. foundational-run — challenged and approved all 39 current-state items; caught a planted SLA contradiction and a RACI conflict and reworked them.
  4. source-regulation / source-cx / source-innovation — web-sourced 7 regulations, 6 competitor-CX + 2 CX benchmarks, 3 trends + 3 competitors + 4 ideas.
  5. source-target + transformation-agent — consolidated a Target Process stub and refined it: 4 target states, 7 transformation decisions, 4 gaps.
  6. run-lint — 8 findings (3 conformance, 4 discrepancy, 1 question); 12 approvals re-opened.
  7. comment + comment-review — posted a comment on PS-CAC-001, ran AI comment resolution; the comment was incorporated and the thread closed.
  8. 2nd document + conflict-resolution — uploaded an update memo that lowered a threshold (EUR 250k → 100k); 6 conflicts flagged and all 6 resolved.

The chat-to-document wiring worked well: after every turn element cards flipped to "Approved", nav status dots went green, and area counts updated (As-Is Process 0/26 → 26/26).

Issues found — 7 feedback items

Filed as FB-002…FB-008 in feedback/; visible in the in-app App Feedback page.

FB-002 Bug

Scope-lock decline points to a CLI skill, not the in-app button

Asked to create a new process from a process-scoped chat, the assistant correctly declined but told the SME to "start a fresh session … and use the /new-process skill" — a CLI instruction. It should point to the + New process button in the top bar.

FB-005 Bug — significant

comment-review & conflict-resolution edit elements but leave the approval stamped

Both skills patch element blocks (resetting block provenance to proposed) but leave the frontmatter approval: approved / approvalBy in place — even though their own chat replies say the element will need re-confirmation. SKILLS.md §10 requires any edit to revert approval to in-progress. Result: in an audit-facing tool, elements show "Approved by M. Berger" over content changed after that approval.

FB-004 Bug

Chat transcript and session are lost on a page reload

Mid-foundational-run, a reload reset the chat to a "Welcome back" banner — the whole transcript and the --resume session id were gone. Run state survived on disk (review-state.json), but conversation continuity did not. The transcript is ephemeral React state with no persistence.

FB-003 Improvement

document-ingest runs as one long turn with no progress signal

Extraction took ~25–30 min for 38 elements. For the first ~10 min nothing was on disk and the activity line sat on a generic "Searching files …" — indistinguishable from a hang. The longest operation in the app has the least progress visibility.

FB-008 Improvement

Element/document view only refreshes when a chat turn ends

The view reacts correctly at turn boundaries but never mid-turn. During the 25-min ingest, 38 files were written one by one while the document view showed an empty process the whole time. Streaming element-written events would make a long turn visibly progress rather than visibly hang.

FB-006 Bug — minor

transformation-agent close-out message contradicts itself

The summary said "Everything is status: draft — review and approve the elements in the web app" while the same message's table marked all 15 elements "✓ M. Berger" — and on disk all 15 were already approved. The boilerplate close-out doesn't reflect what the interactive run actually did.

FB-007 Improvement

Web-sourcing progress is only visible on the launching section

source-* runs show a banner on the section that started them, but nothing global. A multi-minute run is invisible if the SME navigates away. A small top-bar busy indicator would help.
