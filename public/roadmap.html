Processminer Feature Roadmap

Version: 2026-05-28 · Status: Candidate pipeline, not a commitment · Horizon: 2026 H1 → 2028+ · Updated after the v0.4.0 architecture rollout, the 2026-05-28 dogfood run, and the Sparx/Adonis/EPM landscape discussion.

Processminer started as an LLM-Wiki for documenting a single banking process end-to-end with SME interviews. The v0.4.0 rollout finished the foundational engine: derived per-type schemas, a pure-data ProcessView join layer, and an orchestrator state machine that picks the next best action from the wiki's own state. This roadmap is the bridge from "usable for a single dogfood SME" to "institutional bank-wide tool that integrates into the existing landscape and supports the recurring deliverables a process / architecture / change team actually produces".

Bank landscape — the tools we live next to

Processminer does not enter an empty room. The bank already runs four adjacent systems and one shared taxonomy. Every Phase 1 / 6 / 7 idea is shaped by them.

Sparx Enterprise Architect

L2/L3 architecture repository

Holds capability map, application portfolio, integration architecture at strategic abstraction. Does NOT hold L4 detail. Processminer aggregates upward via XML export (Phase 1 #3).

Adonis NP

Partial L4 process repository

Process step lists + owner roles where adopted. Patchy adoption, content often years old; CSV export with varying format. Used as warm-start input (Phase 1 #2).

EPM

Enterprise Process Model — shared taxonomy

3–4 level hierarchy: Process Group → Process Area → Process → Sub-Process. The navigation rückgrat for Processminer (Phase 1 #1). Without it, the three tools cannot stay aligned.

Confluence

Product & tech documentation

Where product and engineering teams already live. Two-way connector (Phase 6 #1) prevents parallel knowledge-base divergence.

Jira

Work tracking

System of record for "what is being done". Pain Points / Innovation Ideas / Gap Resolutions flow forward to Jira epics (Phase 6 #2, Phase 7 #1).

Positioning principles — what Processminer is NOT

The durable framing this roadmap inherits. Every candidate is sanity-checked against these four rules.

1Processminer is the L4 elicitation engine.

It does what Sparx, Adonis and GRC tools structurally cannot: pull tacit per-step detail out of the SME's head into a structured wiki via AI-led interviews. That is the wedge.

2Processminer is NOT the system of record for governance.

Adonis handles process versioning for L4; Sparx handles L2/L3 governance; GRC handles compliance attestation. set_approval.py covers elicitation sign-off, not governance gates. Phase 1 #8 / #9 deprioritised for this reason.

3Processminer aggregates upward and is aggregated into.

Adonis CSV flows IN (warm start). Sparx XML flows OUT (architectural aggregation). Confluence ↔ Jira flow BOTH WAYS (Phase 6). Participant in the bank's landscape, not a replacement.

4The Karpathy wiki is sacred.

Every wiki mutation goes through a schema-validated writer; new orchestration state lives in sidecars outside the wiki tree. No candidate may bypass the writer or invent wiki-internal sidecars.

Prioritisation framework

Every candidate gets one of four tags. The tags are an honest commitment signal, not a wishlist.

TagMeaning
P0Do now. Committed to the current quarter. Gets a *-PLAN.md design doc next.
P1Strong yes, sequence later. Committed in principle, queued for a later quarter.
P2Maybe. Revisit at the next quarterly planning round.
dropNot the right move. Kept in the doc with reasoning so the rationale stays visible.

A typical quarter lands 4–6 P0 ideas (≈1 month of focused work each). Anything above 6 P0 is overcommitment.

Effort sizing: XS 1–2 days · S ~1 week · M 2–3 weeks · L ~1 month · XL multi-month

Phase 1 — Processminer component

15 candidates · SME-facing elicitation + bank-landscape integration
Refined 2026-05-28 with bank-landscape context. Three foundational ideas added at the top reflect the actual setup: Sparx as L2/L3 architecture repository, Adonis (patchy, legacy, CSV export) as partial process source, and a bank-wide EPM as shared taxonomy. Processminer is repositioned as the L4 detail authority that aggregates upward to Sparx and pulls warm-start content from Adonis. Ideas #8 (versioning) and #9 (approval chains) deprioritised — Adonis/Sparx already cover governance.

Phase 2 — ArchitectMiner

20 candidates · architect's tooling around the seven target-architecture sections

ArchitectMiner takes a documented As-Is process and develops the seven sections: capability, target-application, adr, target-integration, component, nfr, migration-phase. Two specialist skills (/domain-architect, /solution-architect) elaborate them. Phase 2 sharpens everything around them.

Visualization
1

Capability map

M

What Radial or matrix view of capabilities + the target applications hosting them. Heat-coded by maturity / coverage / priority.

Why Capability inventory exists as a list; spatial view exposes redundancy and gaps at a glance.

2

Target application landscape diagram

M

What Apps as nodes, integrations as edges, NFR criticality as node size. Extends ProcessFlow swimlane pattern.

Why Integration list is a tabular blob; picture answers "what depends on what" in seconds.

3

As-Is → Target system mapping diagram

M

What For each documented SYS-*, show which target app replaces / consolidates / wraps it.

Why The migration story IS this mapping. ADRs reference it constantly but it lives only in prose.

4

ADR decision graph

S

What ADRs as nodes; "supersedes" + "depends on" as edges. Find orphans and contradictions.

Why ADRs accumulate over years; relationships invisible in a flat list. Architects can't see what's load-bearing vs deprecated.

Modelling & analysis
5

Capability coverage gap analyzer

S

What Flag capabilities not yet mapped to a target app; flag those duplicated across multiple target apps.

Why Obvious "what's missing" check currently a manual eyeball exercise.

6

NFR compliance pass

L

What Validate target architecture actually meets each NFR (e.g. P99 ≤ 200ms against an integration chain with 5 sync hops → flag).

Why Highest-value architect feedback loop. NFRs written aspirationally; nothing checks whether architecture lands them.

7

Integration complexity score

XS

What Per target app, count fan-in/fan-out integrations; flag > N as architectural risk.

Why The biggest failure mode is the central super-app. A score makes it visible.

8

Cross-process capability alignment

M

Why Enterprise architecture is multi-process; today each process plans its target in isolation.

Decision conflict detector

M

Standard methodology support

M

What Extract decisions from process-step bodies — "if amount > EUR 5m → Head of Trade Finance" — into proper Decision Model and Notation tables.

Why Banking is rules-heavy; today rules are buried in step narratives where they can't be audited, tested or exported. DMN makes them first-class.

S

Process maturity assessment

M

What CMMI-style scoring (0–5) across documentation, control coverage, KPI presence, automation, governance. Persisted history.

S

What Per process: owner, escalation chain, last reviewed, next review due, audit cycle, change frequency, automation %. The process owner's daily view.

Why Data exists across wiki + sidecars; never consolidated. One dashboard tile the owner can scan in 30 seconds.

Document cross-check module (Policies / DTPs / KOPs)

L

What Cross-check Policies, DTPs (Detaillierte Tätigkeitsbeschreibungen) and KOPs (Konzern-Operativ-Prozesse) against the documented process. Forward: wiki flags a Policy / DTP / KOP gap as an audit-finding. Backward: wiki suggests Policy / DTP / KOP amendments based on SME elicitation.

Why Today the bank maintains wiki AND policies AND DTPs/KOPs as parallel knowledge bases that drift apart. Audit findings then materialise as "documentation gap" — exactly the gap this module closes. High auditor-visibility win.

Touches new /cross-check-policies skill; new ingestion path for Policies / DTPs / KOPs as raw-sources; new finding kind (policy-mismatch); resolution workflow.

Open strategic questions

Unresolved decisions affecting sequencing. Best resolved before the first P0 set goes into design docs.

  1. Sparx XML flavour. XMI (UML profile) vs ArchiMate Open Exchange. Affects Phase 1 #3 effort sizing (M vs L). Depends on how the bank's Sparx is configured — EA team should confirm.
  2. Adonis CSV variants. No two Adonis exports look the same. Does Phase 1 #2 ship with N field-mapping profiles, or one configurable importer with profile-as-YAML? Configurable is more durable.
  3. Confluence space strategy. Phase 6 #1 — single bank-wide space, one per business unit, or one per process? Affects access control + URL stability.
  4. Jira project mapping. Phase 6 #2 — when Processminer creates an epic, which Jira project does it land in? Per-process default? Per business unit? User-selected each time?
  5. Phase 7 module sequencing. Change / Transformation vs Training vs Document Cross-Check — which ships first? Document Cross-Check is highest auditor visibility; Change / Transformation is highest budget visibility; Training is lowest effort.
  6. Phase 5 trim. The 20 top-architect candidates have not yet been trimmed by the user. A "best 6 to invest in" cut would clarify the 2028 outlook.
