Deep dive · Acceleration item #4

The document-ingest write loop — is the Python spawn the problem?

The original claim: "dozens of sequential Python spawns — 30 elements ≈ 90 spawns — directly targets the many-minutes runs." I measured it. The premise is half wrong, and the fix is right for a different reason.

1 · The measurement

Before designing anything, I timed the actual scripts on this machine (Python 3, the real scripts/wiki/ scripts against process cob-003):

WhatMeasuredNote
Bare Python interpreter start~10–20 ms python3 -c pass
Import wiki_lib + load_schema() ~20 msparses the 61 KB schema JSON
next_id.py (one spawn)~30–60 ms scans the process for the type's ids
check_conformance.py (one spawn) ~30 ms
The premise does not survive the measurement. A script spawn costs ~40 ms, not the 100–300 ms one might assume. Even the worst case — ~90 spawns — totals ~3–4 seconds. That is not where "many minutes" comes from. Collapsing Python startups would save seconds, not minutes. The Python spawn is not the bottleneck.

2 · What the write loop actually does

document-ingest Step 3 already separates drafting from writing. For a ~30-element ingest, Step 3.4 ("Write") runs, per element:

  1. next_id.py <slug> <type> — a Bash tool call (one Python spawn).
  2. Save the element spec to /tmp/<id>.json — a Write tool call.
  3. write_element.py /tmp/<id>.json — a Bash tool call (one Python spawn).

That is 3 agent tool calls per element — for 30 elements, ≈ 90 round-trips, plus the fixed scripts (reset_manifest, add_source, write_overview, check_conformance, check_evidence, write_ingest_report).

The cost of one round-trip is not the 40 ms Python process. It is:

  • A model step. The model must emit the tool-use block, then read the result and decide the next action. Even a mechanical write is a generation step.
  • Context growth. Every tool result ("wrote …/PS-DCR-007.md (created)", an id string) is appended to the conversation. 90 results accumulate — and a warm worker carries that context into every later turn (cf. acceleration #3/#5).

3 · Where the "many minutes" really goes

CostScale (30-element ingest)Batch writer fixes it?
Model extraction reasoning — drafting each element's blocks, then the claim-by-claim verification in Step 3.3 Minutes — the dominant cost No — intrinsic; the model still has to think
Agent tool-call round-trips — ~90, each a model step + a context-bloating result Tens of seconds to ~1–2 min Yes — collapses ~90 → ~2
Python process spawns ~3–4 s total Yes, but the saving is negligible
Reframed verdict. A batch writer is worth building — but not because Python startup is slow. Its real value is collapsing ~90 agent round-trips into ~2, which removes tens of seconds of model-step overhead and stops 90 tool results from bloating the conversation. It will not fix the dominant cost — the per-element extraction-and-verification reasoning. Honest expectation: a meaningful trim, not a transformation.

4 · The proposed batch writer

One script — write_elements.py — takes a single manifest of N elements and does the whole write phase in one process: one schema load, one disk scan, ids assigned in sequence, cross-references resolved, all N files written. 

# manifest.json — the model produces this once, then runs ONE script
{
  "slug": "debit-card-replacement",
  "source": "card-replacement-spec.pdf",
  "elements": [
    { "tempKey": "step-1", "type": "process-step",
      "title": "Report the lost card", "confidence": "high",
      "blocks": [ … ], "provenance": { … } },
    { "tempKey": "role-1", "type": "role", "title": "Card Operations",
      "relations": { "raci": ["@step-1:R"] }, // ← temp ref
      "blocks": [ … ] }
  ]
}

The script runs three in-process passes:

  • Pass 1 — assign ids. Scan the process once for the current max sequence per type, then assign each new element its id with an in-memory counter (no per-element disk re-scan). Build a tempKey → real id map. Updates keep their existing id.
  • Pass 2 — resolve references. Replace every @tempKey in relations with the real id from the map. This is what makes batching safe: the model never has to know an id before it is assigned — it refers to siblings by temp key.
  • Pass 3 — write. Serialise and write each element file, reusing write_element.py's exact serialisation, and record each in the run manifest (log_write) just as today.

Net round-trips for the write phase: the model emits one Write (the manifest) and one Bash (write_elements.py) — 2 calls, not ~90. Conformance is unaffected: document-ingest already runs check_conformance.py once over the whole process at Step 6.

Pros & cons of the batch writer

Pros

  • ~90 agent round-trips → ~2 for the write phase.
  • One tool result instead of 90 — far less context bloat.
  • Ids assigned correctly in one pass; no "reserve the id" dance.
  • Cross-references resolved deterministically via temp keys.
  • Fits document-ingest's existing draft → verify → write phase split — no recovery-safety loss there.
  • Same win available to the autonomous source-* skills, which also write many elements in one non-interactive run.

Cons

  • Does not touch the dominant cost — per-element extraction + verification reasoning.
  • A new script + a new manifest format to maintain and test.
  • A malformed manifest fails as a batch — the model must re-emit; per-element writes fail one at a time.
  • Temp-key resolution is new logic — a dangling @ref must fail loudly, not silently.
  • Wrong fit for the interactive specialists — see §5.

5 · Where it applies — and where it must not

  • document-ingest — yes. It already drafts all, verifies all, then writes all; the write phase is a clean loop to collapse. No SME is in the loop, so there is no per-element approval to preserve.
  • The source-* skills — yes. Non-interactive web-sourcing runs that emit many draft elements at once — the same batch shape applies.
  • The interactive specialists — no. process-specialist and the others write each element the moment the SME approves it (Y/E/R), on purpose: it is recovery-safe and keeps the SME in a per-element loop. Batching their writes would break both. The batch writer is an ingest/sourcing tool, not a universal replacement for write_element.py.

6 · Implementation plan

Phase 1 The manifest schema ~½ day

Define the manifest JSON: per-element tempKey (optional), the existing write_element.py spec fields, and the @tempKey reference convention for relations. Document it next to the script.

Phase 2 write_elements.py ~1 day

The three-pass script. Reuse wiki_lib's serialisation and next_id's id logic — refactor the shared id computation into wiki_lib so both next_id.py and the batch writer use one implementation. Fail loudly on a dangling @ref, an unknown type, or a duplicate tempKey.

Phase 3 Tests ~½ day

Extend test_wiki_scripts.py: a batch of mixed types with cross-references resolves to the same files a sequence of next_id+write_element calls would produce; dangling refs and duplicate temp keys are rejected; the run manifest is recorded identically.

Phase 4 Rewire document-ingest ~½ day

Replace Step 3.4's per-element loop with: assemble one manifest, run write_elements.py once. Keep Steps 3.1–3.3 (draft, field-mapping rules, verification) unchanged — those are the judgement. Then apply the same change to the source-* skills.

Phase 5 Measure ~½ day

Run a real ingest before and after; record wall-clock and round-trip count. Confirm the round-trip collapse is real and the written files are byte-identical to the per-element path.

7 · Risks & open questions

  • It will not fix "many minutes" on its own. If that is the goal, the per-element extraction/verification reasoning is the real target — a different, harder problem than batching writes.
  • Batch failure modes. A manifest that fails validation fails wholesale; the script must report which element and why, so the model can re-emit precisely.
  • Reference integrity. Temp-key resolution is the one genuinely new piece of logic — dangling or duplicate keys must be hard errors, never silent.
  • Updates vs. creates. An element the document refines keeps its existing id (no temp key); the manifest must carry both shapes and the script must not mis-assign an id to an update.
  • One id implementation. Phase 2 should consolidate the id computation so next_id.py and the batch writer cannot drift apart.
