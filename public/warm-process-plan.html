Performance · Process Assistant

Eliminating the per-turn cold start

Every chat turn boots a fresh claude CLI process before the model does any work. This is the largest fixed latency cost in the assistant. Below: the problem, two ways to fix it with pros and cons, a recommendation, and a phased implementation plan.

1 · The problem

The Process Assistant chat is backed by the local claude CLI. Each turn, /api/session (src/app/api/session/route.ts, line 88) calls spawn("claude", …) — a brand-new process, every single message.

What happens on each turn, in order:

POST /api/session
  spawn("claude") ........ new OS process, Node + V8 startup
  discover skills ......... scan ~20 .claude/skills/*/SKILL.md
  init MCP servers ........ connect configured MCP servers
  load config + auth ...... read the Claude Code login
  --resume <id> ........... reload + replay the prior transcript
  model turn begins ...... only now does real work start

Everything in orange is fixed overhead paid before the model produces a single token. An elicitation session is dozens of turns long, so this cost is paid dozens of times — and it is the same whether the turn is a one-word reply or a long extraction. It is also the most likely cause of the "sometimes slow" variability: the work is constant, but a cold start lands on top of it every time.

The goal: pay the skill-discovery / MCP-init / auth cost once, keep a process warm, and feed each turn into it — so a turn starts at "model begins" instead of "spawn".

2 · What the cold start actually costs

The overhead is not one thing — it is a stack of independent costs, each small, that compound:

  • Process & runtime startup — a new OS process, Node bootstrap, V8 warm-up, module loading.
  • Skill discovery — the CLI scans .claude/skills/ and parses the frontmatter of ~20 SKILL.md files to build the skill registry.
  • MCP initialisation — any configured MCP servers are launched and handshaked on every boot.
  • Config & auth — the machine's Claude Code login and settings are read and validated.
  • Transcript replay — --resume reloads the prior session and re-feeds it; this grows with the conversation.

Only the last item is intrinsic to "continue this conversation." The first four are pure boot tax that a warm process pays once.

3 · The two approaches

Approach A  Agent SDK streaming-input session

Replace the per-turn spawn with the Claude Agent SDK running a long-lived streaming-input session: one session object stays alive in the Next.js server process and each turn is pushed into it as a message. Skills are discovered once, MCP connects once, and the session holds conversation context in memory — no --resume replay.

Pros

  • Removes the entire cold start after turn one.
  • Officially supported, first-class API — not a process trick.
  • Native event streaming — dovetails with the reply-streaming setting just shipped.
  • No --resume transcript replay; the session holds context directly.
  • Cleaner long-term architecture than driving a CLI.

Cons

  • Auth risk: must confirm the SDK can use the machine's Claude Code login — the "no API key" rule is a documented design decision (SKILLS.md §2).
  • Larger rewrite of route.ts — it becomes stateful.
  • The server process now holds N live sessions in memory.
  • A new dependency and a shift away from "the CLI runs the skills."
  • Sessions die with the server process — no persistence across restarts.

Approach B  Pooled warm CLI workers

Keep one (or a small pool of) claude CLI processes warm — started once with --input-format stream-json --output-format stream-json and left running. Each chat turn is written to a worker's stdin and its events streamed back. A warm worker already has skills, MCP and auth loaded.

Pros

  • Removes the cold start — paid once at warm-up.
  • Still the same claude CLI — provably keeps the "no API key" model (it is literally today's runtime).
  • Smaller conceptual change; the CLI's streaming-input mode is built for exactly this.
  • Can pre-warm a worker on server start so even turn one is fast.
  • No new SDK dependency.

Cons

  • Long-lived child processes to manage — zombies/leaks if mishandled.
  • One worker handles one conversation at a time — needs a pool + session→worker mapping.
  • Crash/health handling: a worker dying mid-turn must be detected and recycled.
  • route.ts still becomes stateful — same as A.
  • A warm process driven over stdin is more fiddly to reason about than discrete spawns.

4 · Side by side

Dimension A — Agent SDK B — Warm CLI pool
Removes cold start Yes Yes
Keeps "no API key" guarantee Needs verification Guaranteed

Process-lifecycle burden SDK-managed Hand-managed pool

5 · Recommendation

Start with Approach B (warm CLI pool). It is the only option that provably preserves the "no API key — use the machine's Claude Code login" rule, which SKILLS.md §2/§14 records as a settled decision. It is a moderate, well-contained change and reuses the stream-json protocol the route already speaks.

Approach A is the cleaner destination and worth adopting if a quick spike confirms the Agent SDK can authenticate with the existing Claude Code login (no API key). Treat that spike as Phase 0 below: if it passes, A becomes viable; if not, B stands on its own. Either way, the route.ts rework (statefulness, session→worker mapping, idle eviction) is shared, so the work is not wasted.

6 · Proposed implementation plan

Phased, for Approach B — each phase is independently shippable.

Phase 0 Measure & de-risk ~½ day

Instrument the current route: log the time from spawn to the first model event. This gives a hard baseline and confirms the win is worth the work.

In parallel, spike Approach A's open question: can the Agent SDK run on the machine's Claude Code login with no API key? The answer decides whether A stays on the table.

Phase 1 A SessionWorker abstraction ~1 day

Wrap a single long-lived claude child — started with --input-format stream-json --output-format stream-json — behind a small class exposing send(message): AsyncIterable<event>. It owns one process and one conversation; it parses the stdout event stream and isolates turns.

Phase 2 A worker registry / pool ~1 day

A module-level map of appSessionId → SessionWorker in the Next.js server. Create a worker on a session's first turn; evict after an idle timeout; cap the pool size. This relies on the app's single-process local deployment (already a given — the app spawns claude on the same machine).

Phase 3 Rework /api/session ~1 day

Replace spawn-per-request with: look up or create the session's worker, write the turn to it, stream its events back as SSE. Keep the existing event protocol unchanged — progress / delta / done / error — so the client needs no changes. restartSession recycles the worker.

Phase 4 Lifecycle hardening ~1 day

Health checks; restart a worker that crashes; surface a crash mid-turn as a clean error event; kill idle workers; clean up all workers on server shutdown. Pre-warm one worker on server start so even the first turn is fast.

Phase 5 Verify ~½ day

Re-measure against the Phase 0 baseline; confirm turn-start latency dropped. Regression-test a full skill run (document a process, run lint, a foundational-run resume) to be sure a warm process behaves identically to a fresh one.

7 · Risks & open questions

  • Statefulness. /api/session becomes stateful — live workers do not survive a server restart. Acceptable for a local internal tool, but it is a real change from today's stateless route.
  • Zombie processes. Long-lived children must be reliably killed on idle and on shutdown, or they leak.
  • Concurrency. One worker = one turn at a time. The chat UI already blocks input while a turn is pending, so same-session concurrency cannot happen; different sessions get different workers. The pool cap bounds total processes.
  • Session identity. Today's --resume <id> is replaced by "this worker is the conversation." The mapping from the app's chatSessionId to a worker must be exact, and the unscoped new-process flow needs its own worker.
  • Approach A auth (if pursued). The whole case for A hinges on the SDK using the Claude Code login with no API key — verify in Phase 0 before committing.
