ADR-PR-007

Case orchestration via Camunda 8 BPMN engine

Proposed
Domain Case Management Owner M. Holzhauser · Domain Architect Updated 2026-05-19 Reviewers 2/3

Context

✓ human-confirmed · traced to TD-PR-002
Periodic Review cases today live in mailboxes and shared spreadsheets. The Target Process requires a single, stateful case object whose lifecycle is observable end-to-end and whose SLAs can be enforced by the orchestrator rather than by reviewers. The orchestrator must run BPMN authored by the business and integrate with the existing IAM, DMS and DWH.

Decision

✓ human-confirmed
Adopt Camunda 8 (Zeebe + Operate + Tasklist) as the case-orchestration engine. Deploy the SaaS region (eu-frankfurt) for the corporate-clients tenant. Workflows are authored as BPMN 2.0; case state is held by the engine, not the database. Custom UI for reviewers consumes Tasklist + GraphQL.

Alternatives considered

▲ drafted by domain-architect agent · awaiting review
  • Build on existing Pega platform — rejected: licence covers Retail only, corporate tenant would need a separate cluster (cost) and Pega's BPMN export is not round-trip safe.
  • Workflow on top of Salesforce Service Cloud — rejected: review SLA logic does not fit Service Cloud's case escalation model.
  • In-house orchestrator on Postgres — rejected: 4–6 FTE-quarter build, no business-readable model.

Consequences

✓ human-confirmed
  • New runtime dependency on Camunda SaaS — adds NFR-PR-004 (RTO ≤ 4h via active-passive).
  • Reviewers gain a real worklist; manual mailbox triage (PP-PR-002) goes away.
  • IAM integration via OIDC — couples ADR-PR-007 to ADR-PR-003 (single sign-on via Keycloak).
  • BPMN models become a versioned artefact owned jointly by process owner and architect.

Outbox + CDC MESSAGING used in 7 ADRs
Write domain events to a Postgres outbox table inside the same transaction as the business write, then publish via Debezium / Kafka. Guarantees at-least-once delivery without distributed transactions.
When to use
  • Need atomicity between DB write and event publish
  • Existing Postgres infrastructure
  • Cross-system state replication
When not
  • NoSQL primary store
  • Sub-100ms latency requirement
Used by: ADR-PR-008 · ADR-DDM-004 · +5
Saga (choreography) MESSAGING used in 4 ADRs
Long-running business transaction implemented as a chain of local transactions, each publishing an event that triggers the next. Compensating events on failure.
When to use
  • Multi-system workflow with rollback needs
  • No central orchestrator available
When not
  • Cycle detection / tracing is critical (use orchestration)
  • Single-system flow
Used by: ADR-DDM-007 · ADR-FR-003 · +2
CQRS DATA used in 3 ADRs
Separate read and write models. Write side publishes events; read side projects them into denormalised stores tuned for query patterns.
When to use
  • Read load >> write load
  • Multiple distinct read shapes
When not
  • Eventual consistency unacceptable
  • Simple CRUD workload
Used by: ADR-PR-009 · ADR-SPP-002 · ADR-DDM-006
API Gateway + BFF INTEGRATION used in 9 ADRs
Single entry point (Kong / Apigee) handles auth, rate-limit, routing; per-channel BFF (backend-for-frontend) handles channel-specific composition.
When to use
  • Multiple channels (web, mobile, branch)
  • Need uniform auth / observability
When not
  • Single internal consumer
Used by: ADR-PR-001 · ADR-DDM-001 · +7
Idempotency key INTEGRATION used in 5 ADRs
Client supplies a stable key on every retryable write; server de-duplicates on that key with a TTL.
When to use
  • Retryable operations (payments, mandate ops)
  • Network-unreliable clients
When not
  • Naturally idempotent reads
Used by: ADR-SPP-001 · ADR-DDM-005 · +3
Zero-trust mTLS SECURITY used in 8 ADRs
Every service-to-service call authenticated with mutual TLS; certificates issued by a short-lived workload identity provider (SPIRE / Istio).
When to use
  • Multi-tenant cluster
  • Regulatory pressure on internal traffic (DORA)
When not
  • Single-process monolith
Used by: ADR-PR-010 · ADR-DDM-009 · +6
Circuit breaker OPS used in 4 ADRs
Wrap calls to a flaky downstream in a breaker that opens after N failures, returns a fallback for a cool-off period, then half-opens to probe recovery.
When to use
  • Downstream is vendor or external
  • Cascading failures observed
When not
  • Hard consistency required (no fallback safe)
Used by: ADR-PR-012 · ADR-FR-004 · +2
Strangler fig OPS used in 3 ADRs
Route increasing percentages of traffic from a legacy system to its replacement while both run side-by-side; decommission once 100% cuts over.
When to use
  • Risk-averse legacy migration
  • Independent rollback per traffic slice
When not
  • Hard cutover required by regulator
Used by: ADR-PR-007 · MIG-PR-002 · MIG-DDM-001
