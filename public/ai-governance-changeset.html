TL;DR — five architectural moves

Everything below collapses into five structural changes:

1

New governance/ directory

Top-level, outside wiki/. Holds audit log, lineage, test sets, GPAI evidence, model version pin, runbooks. Karpathy boundary respected.

2

API-call wrapper

Every Anthropic call goes through a wrapper enforcing audit logging + kill switch + model-version pin + region check before the call proceeds.

3

Schema + writer additions

Every element must carry modelVersion / skillVersion / promptHash / schemaVersion; contradiction check before approval.

4

Raw-source intake gate

PII classification required at upload; meta.json stamped alongside the file; intake rejects unclassified uploads.

5

UI surface additions

AI disclosure badges, grounding-score chips, right-to-explanation view, governance dashboard, kill-switch banner, PII classification input.

These are accompanied by 9 new Python scripts and 4 new CI gates.

A.1

governance/ directory at repo root

Closes: Phase 1 #4, #5, #7, #8 · Phase 2 #7 · Phase 4 #1, #4, #5 · Phase 5 #4 (homes for the artefacts)

Issue today Today there is no single home for governance evidence. The audit log doesn't exist anywhere. The model card hasn't been written. The incident runbook lives in someone's head. When the regulator or Internal Audit asks "show me X", we'd have to dig through git history, Confluence, shared drives, and prod logs — and we'd still be missing pieces. The wiki stays clean and product-facing; governance evidence is one folder up and clearly separated. CI prevents accidental mixing in either direction. When a regulator asks "show me your model card", we open one file.

B.1

Audit-log wrapper around every Anthropic call

Blocker

Closes: Phase 4 #1 · Phase 1 #6 (if high-risk) · Phase 5 #5 input

Issue today Every interaction between Processminer and Claude is invisible. Nobody can reconstruct "who used Processminer yesterday, with which inputs, getting which outputs?" If a regulator under DORA Article 17 asks for the root cause of an incident, we have no data. If Anthropic later claims they did or didn't process certain data, we can't disprove it. Under EU AI Act Article 12, if Processminer is classified high-risk we need automatic logging retained for the duration the regulator expects (typically 10 years for bank records) — and we have nothing today.
What changes A thin wrapper sits between Processminer and Anthropic. Every single AI call is recorded — when, by whom, for which process, with what input, what output, which Claude version. Stored as plain-text append-only files, retained 10 years. From this one record, every downstream governance question becomes answerable: incident root cause, GDPR data-flow tracing, MRM monitoring, regulator forensics. The user sees nothing change; the audit team gets everything they need.

B.2

Kill switch feature flag

High

Closes: Phase 4 #6

Issue today If something goes seriously wrong — Anthropic has a security incident, a skill starts producing harmful outputs, a regulator orders an immediate pause — there's no way to stop Processminer short of un-deploying the application. EU AI Act Article 14 explicitly requires the operator of an AI system to have an "interrupt" capability. DORA resilience testing assesses whether the bank can disable a critical ICT service quickly. We can do neither today.
What changes A central switch — flipping a flag in a config file or environment variable — immediately stops every AI call across the bank within seconds. Users see a clear in-app banner explaining that the system is paused. The switch is tested at least once a year so we know it works. When a regulator asks "can you stop?", the answer is "yes, and here's the test log proving it does."
Engineering detail
NEWsrc/lib/feature-flag.ts reads PROCESSMINER_DISABLED env + governance/disabled.flag
MODsrc/app/api/session/route.ts return 503 + explanation when disabled
MODsrc/lib/anthropic-wrapper.ts refuse upstream call when disabled
NEWsrc/components/SystemDisabledBanner.tsx bank-wide banner
MODsrc/app/layout.tsx render banner when flag set
B.3

Anthropic region + model version pin check

Blocker

Closes: Phase 5 #2 (residency) · Phase 7 #2 (version pin)

Issue today Claude has versions and runs in regions. Nothing checks which version we are calling or which region the request goes to. Anthropic could route a request to a US data centre on a Wednesday afternoon — and the bank would have unknowingly transferred internal data outside the EU, a GDPR Chapter V violation. Anthropic could push what they call a "minor" model update that meaningfully changes how skills behave — and we'd only notice when a user reports surprising outputs. Both scenarios are data-protection and model-risk findings.
What changes A small config file declares upfront: "We use Claude version X, in region Y, and nothing else." Every call checks this before sending. Any mismatch is refused and an alert fires. Planned upgrades go through a documented impact review and MRM sign-off before the pin is changed. From the regulator's perspective: we can prove which model version processed which data, and where it physically went.
Engineering detail
NEWgovernance/model-version.json { "pinned": "claude-opus-4-7", "region": "eu-central-1" }
MODsrc/lib/anthropic-wrapper.ts refuse call if region or version drifts from pin
NEWscripts/wiki/check_model_version_pin.py CI check (see E.3)
B.4

Drift / quality signals pipeline

High

Closes: Phase 4 #3 · Phase 2 #6 input

Issue today We have no idea whether Processminer is still working well over time. If skill quality silently degrades — Anthropic's model changed, a prompt drifted, a new edge case emerged — we would notice months later through accumulated bad documentation that SMEs have already signed off on. MRM expects monthly evidence that the model still performs within its validated range. We cannot produce that evidence today.
What changes An automated pipeline calculates quality metrics every day: how often the schema is broken, how often AI-generated citations are missing, how often SMEs heavily rewrite AI output (the best proxy for "the AI got it wrong"). Trends are charted over 30 / 90 / 365 days. Below-threshold values trigger an automatic alert. MRM sees the trend; the Committee sees the trend; the regulator sees the trend.
Engineering detail
NEWsrc/lib/quality-metrics.ts schema conformance, citation coverage, edit rate, lint trend, conflict-resolution rate
NEWsrc/app/api/governance/metrics/route.ts rolling 30-day signals

Source data: governance/audit-log/*.jsonl + existing provenance.json + existing lint.json

B.5

Retention pipeline

High

Closes: Phase 5 #4

Issue today The wiki grows forever. Some content needs 10-year retention (regulatory records), some 3-year (operational documentation), some indefinite, some should be deleted to comply with GDPR's storage-limitation principle. There's no enforcement; old content just accumulates. If a customer or employee invokes GDPR Article 17 "delete my data", we can't cleanly comply — we don't know where their data lives or when it should expire.
What changes A retention policy document declares how long each kind of content stays. A daily automated job applies the policy — moves expired content to a separate "disposed" area with a written disposition log. A legal-hold flag exists so content under investigation is never auto-disposed. When the DPO asks "how do we honour an Article 17 request on this person's data?", the answer is "the retention engine handles it; here's the log of dispositions."
Engineering detail
NEWscripts/wiki/apply_retention.py reads retention-policy.md rules; moves expired to governance/disposed/
NEWcron entry (host-OS / k8s) daily execution
MODschema/process-schema.json optional legalHold: boolean on every element type; retention skips when set
B.6

Vendor SLA monitor

Medium

Closes: Phase 4 #7

Issue today We have no monitoring of whether Anthropic is even up. Outages are discovered through user complaints. There's no historical availability record to feed into the annual DORA Article 29 third-party concentration review. If we ever needed to demonstrate to the regulator that Anthropic met (or didn't meet) agreed SLAs, we'd have nothing.
What changes A small automated job pings the Anthropic API every five minutes from the bank's infrastructure. Each result is logged. From this, we can produce a monthly availability report that feeds vendor risk reviews, the governance dashboard, and any regulatory request about ICT third-party reliability.
Engineering detail
NEWscripts/audit/probe_anthropic.py periodic health check → governance/vendor-sla/<YYYY-MM>.jsonl
B.7

Champion / challenger harness

Medium

Closes: Phase 7 #5

Issue today Today Processminer depends entirely on Anthropic. If Anthropic raises prices 5×, has a structural outage, changes their terms in a way the bank can't accept, or is sanctioned — we have nowhere to go. DORA Article 29 requires the bank to address ICT third-party concentration risk; we can't demonstrate we have an alternative ready.
What changes The code is restructured so that swapping Anthropic for another LLM provider (e.g. an EU-hosted Mistral or Aleph Alpha) is a configuration change, not a code rewrite. A quarterly job runs identical prompts through both providers and compares outputs. We have decision-grade data on how a switch would behave before we ever need to make one.

C. Schema + writer changes (Python + JSON)

5 items · governance frontmatter · PII intake · lineage · contradiction · grounding
C.1

Element governance frontmatter

High

Closes: Phase 7 #6 · enables Phase 2 #6 + Phase 5 #5

Issue today When an SME, auditor, or regulator looks at a documented process, they cannot tell which version of Claude wrote it, which version of the skill was used, or with which prompt. If the regulator asks "reproduce this output six months later under the same conditions", we cannot. If a skill update later breaks something, we cannot tell which content was authored under the old behaviour versus the new. MRM cannot enforce annual revalidation because we don't know what was generated by what.
What changes Every documentation element carries a small machine-readable stamp at the top — Claude version, skill version, prompt fingerprint, schema version, generation timestamp. From this stamp, every output is reproducible. "What produced this?" becomes a single query. MRM's annual revalidation has the data it needs.
Engineering detail

Every element type gets these fields in its frontmatter contract:

modelVersion: claude-opus-4-7@20260301     # which Claude generated this
skillVersion: sha256:abc...               # SHA of the skill's SKILL.md
promptHash: sha256:def...                 # SHA of the rendered prompt
schemaVersion: 0.4.0                        # schema version at write time
generatedAt: 2026-05-28T22:34:00Z           # set once on first author
disclosureLabel: AI-generated              # transparency surface (Art. 50)
MODschema/process-schema.json add to every element type
MODscripts/wiki/write_element.py refuse to write without these
MODscripts/wiki/patch_element.py preserve / update on patch
MODscripts/check_conformance.py ERROR if missing

C.2

PII classification on raw-sources

Blocker

Closes: Phase 5 #1

Issue today Users upload documents — process manuals, audit reports, screenshots, occasionally exports that contain customer or employee personal data. Nothing in the tool today asks "what's in this document?" If a user accidentally uploads a customer-data-laden export, it lands in the system, gets sent to Anthropic, and is now subject to every data-protection finding that follows. GDPR Article 5 data-minimisation requires preventing this proactively, not after the fact.
What changes At the moment of upload, the user must declare what kind of data the file contains: none / internal / customer PII / special category. The system refuses uploads without a classification. The classification then drives downstream behaviour — which Claude region the file can be sent to, what retention applies, how granular the audit log must be. If a customer-PII file slips through, the DPO has a clear record of who declared what.
Engineering detail
NEWraw-sources/<slug>/<file>.meta.json { "piiClass": "none"|"internal"|"customer"|"special" }
MODsrc/app/api/upload/route.ts reject if missing or invalid
NEWscripts/wiki/check_pii_classification.py fails CI if any raw-source lacks meta.json

UI side: see D.3.

C.3

Lineage sidecar (outside wiki tree)

High

Closes: Phase 5 #5 · Phase 5 #6 input

Issue today The existing provenance contract tells us "where does this heading come from" at the source- document level. It doesn't show the full chain: who triggered the session, which Claude version, which skill version, which source documents were in scope, which proposals were accepted versus rejected, which approvals were granted. This record is what feeds the right-to-explanation view (D.4) and MRM's reproducibility evidence.

C.4

Contradiction check before approval

Medium

Closes: Phase 6 #7

Issue today Skills can produce content that contradicts other approved content elsewhere in the wiki — for example, the AI proposes "Role X owns step Y" when "Role Z owns step Y" is already approved. There is no automatic check; lint might catch it later but only once explicitly run. If a contradiction slips through approval, the documented process becomes internally inconsistent — which is exactly the kind of finding an external auditor lands on first.
What changes Before any approval is granted, an automatic check runs against the rest of the wiki for structural contradictions (role assignments, control owners, SLA values). If a contradiction is detected, the approval is refused with a clear explanation of what conflicts with what. The reviewer either reconciles the conflict or explicitly overrides with a documented reason. The inconsistency-finding-by-auditor failure mode is closed.
Engineering detail
MODscripts/check_conformance.py add contradiction rules: role assignment conflict, control-owner contradiction, SLA conflict
MODsrc/lib/conformance.ts parity
MODscripts/wiki/set_approval.py hard-fail if contradiction; require --override <reason>
C.5

Source-grounding score

High

Closes: Phase 6 #6

Issue today The provenance contract says "every derived heading must cite an evidence quote from a source document". Today this is an aspiration; nobody actually measures compliance. If the AI hallucinates a citation that doesn't exist in the source, we would only discover it manually, one element at a time. There's no quantified evidence MRM or the regulator could see proving the provenance contract works.
What changes An automated check walks every element, finds each cited quote, and verifies it actually exists in the source document via deterministic text match. The result is a score (0–100%), shown as a colour-coded chip on each element card (D.2). Low scores trigger investigation; aggregate trends feed the quality dashboard. The aspiration becomes a measurable, auditable number — exactly what AI Act Article 15 evidence requires.
Engineering detail
NEWscripts/wiki/score_grounding.py % of derived/source headings with citable evidence that survives deterministic match
NEWwiki/processes/<slug>/grounding.json derived sidecar; stays inside wiki/ (existing precedent: lint.json, raci.json)
MODsrc/components/ElementCard.tsx surface chip — see D.2

D.1

AI disclosure across surfaces

High

Closes: Phase 1 #3 (Article 50)

Issue today Under EU AI Act Article 50, when a user interacts with AI or reads AI-generated content, they have a right to know. Today none of Processminer's surfaces say "this was AI-generated". A user reading an approved element might reasonably assume a human wrote it. A Confluence page produced by Processminer would look like any other Confluence page. It appears on element cards, in the chat header, on PDF exports, on published Confluence pages, and as an XML attribute on Sparx exports. Small but unmistakable. When the regulator asks how we comply with Article 50, we point at the badge.

D.4

Right-to-explanation view

High

Closes: Phase 5 #6 (GDPR Art. 22 / AIA Art. 86)

Issue today Under GDPR Article 22 and EU AI Act Article 86, when AI-influenced decisions affect a natural person, the person has a right to a meaningful explanation of how the decision was reached. Even with all the underlying data we collect (provenance, audit log, run manifest), we cannot produce a human-readable explanation today. Accessible to authorised users (Compliance, DPO, Internal Audit, the affected person via the data-subject-access process). One page, plain language, regulator-ready.

E.2

CAB ticket required on skill / schema changes

High

Closes: Phase 7 #1

Issue today A change to a skill or schema is a change to the AI model. MRM treats it as a model change that requires a documented impact review through the bank's Change Advisory Board (CAB) process. Today engineers can change skills with just a code review; the CAB process isn't enforced. MaRisk AT 7.2 expects this enforcement.
What changes An automatic CI check on every pull request that touches the skills or schema folders requires a CAB-NNNN ticket reference in the PR body. No CAB ticket → no merge. The CAB process is now mechanically enforced rather than relying on engineer memory.
Engineering detail
# .github/workflows/cab-gate.yml (new)
on:
  pull_request:
    paths:
      - '.claude/skills/**'
      - 'schema/**'
      - 'governance/model-version.json'
jobs:
  cab-check:
    runs-on: ubuntu-latest
    steps:
      - name: Require CAB-NNNN reference
        run: |
          if ! gh pr view --json body -q .body | grep -qE 'CAB-[0-9]+'; then
            echo "::error::PRs touching skills/schema require a CAB-NNNN reference"
            exit 1
          fi
