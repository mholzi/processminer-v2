The **phantom-tool program**: a sweep that closed the gap between what the
skills *called* and what the backend *exposed*. Before this, ~14 tools the
`.claude/skills/*` prompts instructed the AI to call had **no definition in
`src/`** — so the affected skills could not complete their writes against the
real backend (and `/dogfood-run` failed). After PR #44 the skills' tool-call set
**exactly matches** the registered tools: **19 tools, both providers, drift-free,
zero phantoms**.

Shared mechanics across the program:
- Every tool is registered in **both** providers — `gemini-worker.ts`
  (`toolDeclarations`) and `claude-mcp-server.ts` (`ListTools` + handler) — kept
  symmetric so behaviour is provider-agnostic.
- Write logic lives in **pure, unit-tested helper modules** (`session-*.ts`); the
  provider handlers stay thin (read JSON → call helper → write JSON).
- **Runtime/orchestration state never enters the wiki JSON** (Karpathy
  guardrail) — it lives in the runtime store (`data/runtime/<slug>.json`).

## PR #39 — `scaffoldProcess`

**Issue.** The `new-process` skill called `scaffoldProcess({ slug, PROC, title,
description })` to create a brand-new empty process document, but the tool did
not exist — the MCP server threw `Process document not found` for any absent
slug. `new-process` was non-functional end-to-end and `/dogfood-run` Stage 1
failed.

**Resolution.** Added a shared `buildProcessDoc(PROC, title, description)` (root
`meta` + empty overview, id `${PROC}-001`) and a `scaffoldProcess` tool in both
providers. The Claude handler runs *before* the "must already exist" guard;
both validate the slug/PROC and refuse to overwrite.

**Changed files.**

new-document write** rather than an element-scoped one — established that the
session tool surface must cover process creation, not just element CRUD.

## PR #40 — `createElements`, kill the run-manifest

**Issue.** The four `source-*` skills and `document-ingest` told the AI to call
`createElements` / `writeElements` plus a family of "run manifest" tools
(`resetManifest`, `mergeManifests`, `getSourceReport`, `generateSourceReport`)
to batch-write elements and tally a report — **none of which existed**. Their

migrated onto the same helper (DRY). The five skills now read their report
counts from that return, so the **entire run-manifest tool family was deleted**.

  `source-target`, `document-ingest`

**Critical architecture impact.** **Eliminated a whole runtime concept (the run
manifest) rather than implementing it** — report counts now derive from the
tool's return value, keeping transient state out of the wiki (Karpathy). Also
established the "single `createElement` and batch share one tested core" pattern.

## PR #48 — Dogfood #1 + #7: SSE heartbeat + atomic process-JSON writes

**Source.** The first two findings actioned from the **2026-06-05 dogfood run**
(`public/test-report-assets/2026-06-05-0726/FINDINGS-REPORT.md`, FRDT-001 ·
*Funds Release Dogfood 2*). Landed together — both are small, self-contained
robustness fixes to the session/write path.

**Issue (#1 — 🔴 most significant).** Three long-running skills (`source-target`,
`control-compliance-specialist`, `run-lint`) dropped the browser's live view
even though the worker kept running server-side and wrote everything correctly.
Root cause: a silent turn (a long sub-agent fan-out, or pure model reasoning)
emits no SSE events, so the client's 5-min stuck-turn watchdog
(`ProcessDocScreen.tsx`) wrongly declared *"Lost contact with the assistant"* on
a healthy turn. A real SME then restarts — which **kills the still-running
worker**, destroying good work.

**Issue (#7 — 🟢 flagged low-confidence, confirmed real).** The dogfood
`cx-benchmarks` count read 3 → 6 → 3 across mid-write disk checks. Confirmed: the
source-of-truth process JSON was written with a plain `writeFileSync`
(truncate + stream-in) at **every** write site, so a concurrent reader could see
a half-written/empty file and a crash mid-write corrupted the document.

**Resolution.**
- **#1:** the route emits a `{type:"ping"}` heartbeat every 20s for the life of
  the turn; the shared `agent-chat` `apply()` already fires `onAnyEvent` on every
  parsed event, which re-arms the watchdog. The ping carries no payload and has
  no UI case, so it is invisible.
- **#7:** new `atomicWriteFileSync` (write a sibling `.tmp` file → `rename` over
  the target, atomic on POSIX) replaces every process-JSON write — a reader now
  sees only the old or new document, never an intermediate. The two non-doc
  writes (sessions-map, debug dump) were deliberately left as-is.

**Changed files.**
- `src/app/api/session/route.ts`, `src/lib/agent-chat.ts` (#1)
- `src/lib/atomic-write.ts` (new), `src/lib/wiki-write.ts` (4 sites),
  `src/lib/claude-mcp-server.ts` (9), `src/lib/gemini-worker.ts` (7),
  `src/lib/runtime-store.ts` (1) (#7)
- `docs/PRs 06062026.md`

**Scope note.** The working tree carried unrelated in-flight work (DTP panel,
schema, UI). This PR is **only** the heartbeat + atomic-write changes — the
atomic hunks in the two workers + runtime-store were staged from `HEAD` via
`git apply --cached`, so none of that WIP is included.

**Verification.** `npm run typecheck` clean; `npm test` **105/105**;
`wiki-write.test.ts` 3/3. The heartbeat's full behavioural proof needs a
multi-minute silent turn (not practically stageable); the mechanism is verified
through the existing `onAnyEvent` watchdog path and the types.

**Critical architecture impact.** None to the data model. Establishes that **all
durable-JSON writes go through one atomic helper** (corruption-safe), and that
the SSE protocol carries a liveness signal so the client can distinguish a silent
healthy turn from a dead one — the precondition for any future
reconnect/resume-the-in-flight-turn work (the deeper part of finding #1, not
attempted here).

## PR #49 — Dogfood #2–#6: skill prompt refinements + tempKey lint guard

**Source.** The next five findings from the **2026-06-05 dogfood run**
(`public/test-report-assets/2026-06-05-0726/FINDINGS-REPORT.md`). Stacked on #48;
all are prompt/doc refinements plus one conformance guard — **no runtime or
provider code touched**.

**Issues & resolutions.**
- **#3 — foundational-run `[E]` stall (🟠).** After an `[E]` edit the worker
  frequently ended the turn without advancing, needing a manual "continue". Made
  **step 5 (Advance) explicit and unconditional** — same turn, after *every*
  outcome — and added a matching reminder in the `[E]` branch.
- **#6 — `new-process` abbreviation round-trip (🟡).** The scaffolder rejects
  digits (`^[A-Z]{2,6}$`) but the skill only said "uppercase", so `FRD2` was
  proposed → rejected → `FRDT`. The skill now states the letters-only, 2–6 rule.
- **#2 — ingest tempKey leaks (🟠).** document-ingest could leave a raw
  cross-reference key (`(ps-1)`) in **prose**; the batch writer only resolves
  standalone `@key` *reference values*, so it leaked. **Prevention:** keep
  tempKeys to relation fields, name elements by title in prose. **Safety net:**
  new `checkTempKeyLeaks()` flags any tempKey left in a block as a lint finding
  (re-opens the element). Its alphabet is the schema's own **idPrefixes**, so it
  matches the ingest namespace exactly — never a real id (`PS-FRDT-001`) nor
  ordinary prose (`tier-1`, `Basel-3`, single-letter `M-1`). +3 unit tests.
- **#4 — free-chat misrouting (🟡).** A named-specialist request was
  re-summarised instead of run. Added a **Skill Routing** block to
  `CORE_SYSTEM_PROMPT`: a message that names a skill runs that skill (named
  skill beats recency); CTAs are the reliable path.
- **#5 — long silent burst-writes (🟡).** `source-cx` and `source-regulation`
  (no cross-refs) now **write incrementally** — a `createElements` call per
  group with a running counts tally — so elements appear as the scan proceeds.
  `source-innovation` and `source-target` **stay single-batch** by design (they
  resolve `@tempKey` cross-references within one batch; splitting would break
  them) — now documented in-skill. #1's heartbeat already covers the
  connection-drop half of this finding; #5 is the visible-progress half.

**Changed files.**
- `.claude/skills/foundational-run`, `new-process`, `document-ingest`,
  `CORE_SYSTEM_PROMPT.md`, `source-cx`, `source-regulation`, `source-innovation`,
  `source-target`
- `src/lib/conformance.ts`, `src/lib/conformance.test.ts`
- `docs/PRs 06062026.md`

**Verification.** `npm run typecheck` clean; `npm test` **106/106** (+3
tempKey-guard tests, incl. the false-positive guard). The prompt changes' real
proof is a future dogfood run — verified here by inspection.

**Critical architecture impact.** None to the data model. Adds one lint
dimension (tempKey leaks), keyed off the schema's idPrefixes so it stays in sync
with the element-type set automatically. The `source-*` split (incremental vs
single-batch) is now an explicit, documented invariant tied to whether a skill
uses `@tempKey` cross-references.

## Net effect on the architecture

- **Tool surface:** 9 → **19** session tools, registered identically in both
  providers (drift-guarded by symmetry).
- **New pure helper modules:** `session-create.ts`, `session-notes.ts`,
  `session-cursor.ts` (+ additions to `session-writes.ts`) — all unit-tested,
  provider-agnostic. Test count **70 → 100**.

  run-manifest concept was removed entirely.
- **Guardrail upheld throughout:** durable process knowledge in
  `wiki/processes/<slug>.json`; runtime/orchestration state (`reviewState`,
  `qerState`, lint, dismissals) in `data/runtime/<slug>.json`.
- **Result:** every skill's tool calls now resolve to a real, schema-enforced
  tool — the skill↔backend drift documented in `BRIDGES_AND_TODOS.md §2D` is
  closed.
