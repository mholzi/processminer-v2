---

The **phantom-tool program**: a sweep that closed the gap between what the
skills *called* and what the backend *exposed*. Before this, ~14 tools the
`.claude/skills/*` prompts instructed the AI to call had **no definition in
`src/`** — so the affected skills could not complete their writes against the
real backend (and `/dogfood-run` failed). After PR #44 the skills' tool-call set
**exactly matches** the registered tools: **19 tools, both providers, drift-free,
zero phantoms**.

Shared mechanics across the program:
- Every tool is registered in **both** providers — `gemini-worker.ts`
  (`toolDeclarations`) and `claude-mcp-server.ts` (`ListTools` + handler) — kept
  symmetric so behaviour is provider-agnostic.
- Write logic lives in **pure, unit-tested helper modules** (`session-*.ts`); the
  provider handlers stay thin (read JSON → call helper → write JSON).
- **Runtime/orchestration state never enters the wiki JSON** (Karpathy
  guardrail) — it lives in the runtime store (`data/runtime/<slug>.json`).

## PR #39 — `scaffoldProcess`

**Issue.** The `new-process` skill called `scaffoldProcess({ slug, PROC, title,
description })` to create a brand-new empty process document, but the tool did
not exist — the MCP server threw `Process document not found` for any absent
slug. `new-process` was non-functional end-to-end and `/dogfood-run` Stage 1
failed.

**Resolution.** Added a shared `buildProcessDoc(PROC, title, description)` (root
`meta` + empty overview, id `${PROC}-001`) and a `scaffoldProcess` tool in both
providers. The Claude handler runs *before* the "must already exist" guard;
both validate the slug/PROC and refuse to overwrite.

**Changed files.**
- `src/lib/gemini-worker.ts`, `src/lib/claude-mcp-server.ts`
- `docs/BRIDGES_AND_TODOS.md`, `docs/PR-doc-realignment.md`

**Critical architecture impact.** First tool that performs a **root-level /
new-document write** rather than an element-scoped one — established that the
session tool surface must cover process creation, not just element CRUD.

## PR #40 — `createElements`, kill the run-manifest

**Issue.** The four `source-*` skills and `document-ingest` told the AI to call
`createElements` / `writeElements` plus a family of "run manifest" tools
(`resetManifest`, `mergeManifests`, `getSourceReport`, `generateSourceReport`)
to batch-write elements and tally a report — **none of which existed**. Their
write+report path was non-functional (the `/tmp`-scratch improvisation seen in
the dogfood run).

**Resolution.** New `createElements` tool (both providers) backed by a shared,
unit-tested `session-create.ts` (`buildElement` + `createElementsBatch`): it
resolves `@tempKey` cross-references across the batch, isolates a failing
element, and returns per-type **`counts`**. The single `createElement` was
migrated onto the same helper (DRY). The five skills now read their report
counts from that return, so the **entire run-manifest tool family was deleted**.

**Changed files.**
- `src/lib/session-create.ts` (new), `src/lib/session-create.test.ts` (new)
- `src/lib/gemini-worker.ts`, `src/lib/claude-mcp-server.ts`, `package.json`
- `.claude/skills/source-cx/SKILL.md`, `source-innovation`, `source-regulation`,
  `source-target`, `document-ingest`
- `docs/BRIDGES_AND_TODOS.md`, `docs/PR-doc-realignment.md`

**Critical architecture impact.** **Eliminated a whole runtime concept (the run
manifest) rather than implementing it** — report counts now derive from the
tool's return value, keeping transient state out of the wiki (Karpathy). Also
established the "single `createElement` and batch share one tested core" pattern.

## PR #41 — Rewrites onto existing tools

**Issue.** Four phantom tools didn't need to exist — the capability was already
present: `writeOverview` / `updateProcessOverview`, `getNextId` /
`generateNextId`, `getElementTemplate` / `getTemplate`, `checkEvidence`.

**Resolution.** Rewrote the skills, no new tool surface:
- overview writes → `updateElement` on the overview's **root id** (it already
  patches root `meta`/`content` when `doc.meta.id === id`);
- `getNextId`/`generateNextId` → dropped (`createElement` assigns + returns the id);
- template lookups → refer to the **schema template**;
- `checkEvidence` → `checkConformance` (its provenance check already verifies
  `document` evidence quotes).

**Changed files.**
- `.claude/skills/document-ingest`, `qer-session`, `add-entry`, `comment-review`,
  `foundational-run` (all `SKILL.md`)
- `docs/BRIDGES_AND_TODOS.md`, `docs/PR-doc-realignment.md`

**Critical architecture impact.** **No code change** — confirmed that several
"missing tools" were really skill drift, and that `updateElement` is the single
write path for both elements *and* the overview. Reduced the tool surface the
backend must carry.

## PR #42 — Root-field tools (`writeIngestReport`, `clearConflicts`; drop `addSource`)

**Issue.** `document-ingest` and `conflict-resolution` called `writeIngestReport`,
`clearConflicts` and `addSource` — none existed. The ingest/triage flow could
not write its result.

**Resolution.** Two shared, unit-tested helpers in `session-writes.ts`
(`buildIngestReport`, `clearIngestConflicts`) + the two tools in both providers,
writing the `ingest` field the triage screen reads. **`addSource` was dropped**:
sources are derived from `raw-sources/<slug>/` + `uploads.json` by `listSources`
(written by `/api/upload`), **not** the process JSON — the skill's "records in
the overview" claim contradicted the model; the skill now just identifies the
already-uploaded file.

**Changed files.**
- `src/lib/session-writes.ts`, `src/lib/session-writes.test.ts`
- `src/lib/gemini-worker.ts`, `src/lib/claude-mcp-server.ts`
- `.claude/skills/document-ingest/SKILL.md`
- `docs/BRIDGES_AND_TODOS.md`, `docs/PR-doc-realignment.md`

**Critical architecture impact.** Clarified the **source-of-truth boundary**:
the `ingest` report is durable process data (in the wiki JSON), but the source
*list* is filesystem-derived (`raw-sources/` + `uploads.json`), never duplicated
into the process JSON.

## PR #44 — Session-cursor + `setApproval` (closes the program)

**Issue.** `foundational-run` and `qer-session` — the two core interactive
interview flows — called a resumable-cursor API (`buildQueue` /
`getSessionStatus` / `advanceSession` / `startSession`) and `setApproval`, none
of which existed. Neither flow could run or resume, and the SME could not sign
an element off mid-walk.

**Resolution.**
- New pure, unit-tested `session-cursor.ts`: `buildFoundationalQueue` (overview
  first, current-state sections in order, process-gaps last; forward-looking /
  target excluded), `newReviewState`, `advance`, `foundationalStatus` /
  `qerStatus`, the `QER_STEPS` sequence, and the canonical
  `FOUNDATIONAL_OUTCOMES_LINE` / `CLOSEOUT_TEMPLATE`.
- `buildApprovalPatch` in `session-writes.ts`; the AI `setApproval` builds an
  approval patch and calls the existing `updateElement`, which **already
  enforces the approval gate** — no new gate logic.
- Five tools in both providers; `getSessionStatus` / `advanceSession` route by
  `kind` (`foundational` default | `qer`).
- New **`qerState`** runtime field, separate from `reviewState`, so the
  orchestrator's foundational-resume read is never confused by a QER session.
- `qer-session` passes `kind: "qer"`.

**Changed files.**
- `src/lib/session-cursor.ts` (new), `src/lib/session-cursor.test.ts` (new)
- `src/lib/session-writes.ts`, `src/lib/session-writes.test.ts`,
  `src/lib/runtime-store.ts`
- `src/lib/gemini-worker.ts`, `src/lib/claude-mcp-server.ts`, `package.json`
- `.claude/skills/qer-session/SKILL.md`
- `docs/BRIDGES_AND_TODOS.md`, `docs/PR-doc-realignment.md`

**Critical architecture impact.**
- **Two separate cursors, two runtime fields.** `reviewState` (foundational
  element walk) stays the field `orchestrator.ts` reads for its resume action;
  `qerState` (QER step sequence) is new and deliberately separate to avoid
  conflating the two on the dashboard.
- **`setApproval` reuses the existing gate** via `updateElement` — the approval
  contract has exactly one enforcement point.
- **Deferred (browser-verifiable follow-up):** a dashboard "resume QER session"
  tile (a new `ActionSpec` kind rendered across ~8 UI components). The tools
  work without it.
- **Verification caveat:** the queue order, the QER step granularity, and the
  canonical `outcomes_line`/`closeout_template` text are read from the skill
  prose and **not live-verifiable** (real foundational/qer runs against process
  data are off-limits). They are plain constants in `session-cursor.ts`, trivial
  to adjust if a mismatch surfaces.

## Net effect on the architecture

- **Tool surface:** 9 → **19** session tools, registered identically in both
  providers (drift-guarded by symmetry).
- **New pure helper modules:** `session-create.ts`, `session-notes.ts`,
  `session-cursor.ts` (+ additions to `session-writes.ts`) — all unit-tested,
  provider-agnostic. Test count **70 → 100**.
- **Runtime store:** gained `qerState` alongside `reviewState`; the
  run-manifest concept was removed entirely.
- **Guardrail upheld throughout:** durable process knowledge in
  `wiki/processes/<slug>.json`; runtime/orchestration state (`reviewState`,
  `qerState`, lint, dismissals) in `data/runtime/<slug>.json`.
- **Result:** every skill's tool calls now resolve to a real, schema-enforced
  tool — the skill↔backend drift documented in `BRIDGES_AND_TODOS.md §2D` is
  closed.
