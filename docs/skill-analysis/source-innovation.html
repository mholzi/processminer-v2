source-innovation

Autonomously sources the innovation perspective from the web — market trends, what EU/global/fintech competitors are doing, and innovation ideas that address the documented problems — writing the whole run as one cross-referenced batch so ideas trace back to their trends and sources.

Non-interactive · button-invoked SOURCE: web · provenance=web FILLS: Market Trends · Competitor · Innovation Ideas Triaged, not approved

a What it does

source-innovation is the web-research analogue of document-ingest: where ingest extracts from an uploaded document, this extracts from the live web. Invoked with a process <slug>, it drafts market-trend elements, the three competitor-move tiers (competitor-eu, competitor-global, competitor-fintech) and innovation-idea elements — each idea linked to the documented pain it addresses.

It does not produce innovation-risks (the innovation-analyst's) or the Target Process (the transformation-agent's) — both need the SME. It is the fast sourcing pass that those deeper interactive skills build on.

Key constraints and behaviours:

  • Non-interactive. No SME, no Y/E/R, no approval — all status: draft for later triage.
  • Reads wider than the happy path. It deliberately reads the control / regulatory angle (control, control-gap, compliance-gap, audit-finding) and the automated/STP branches — a control gap is as real an improvement target as a pain-point.
  • Fills pending citations first. An element with sourceUrl: pending is an analyst draft waiting for citation; it verifies and fills it in place, never duplicating it with a fresh draft.
  • Single cross-referenced batch. Unlike source-cx / source-regulation (incremental), the whole run is written in one createElements batch — ideas reference trends and moves by "@<tempKey>", and the writer only resolves those references within one batch.
  • Coverage gate. It must reach idea coverage "complete": true — every pain-point, friction-point, process-gap and control-gap is addressed by some idea, or the leftover is a noted, conscious gap.
  • Provenance = web. Each block (The trend / Relevance / Evidence) carries source: web with URL + snippet + fetch date, blocking approval until SME confirmation.

b The sequence

Six prescribed steps, ending in one batched, coverage-checked write.

  1. Step 1 — Read the processRead the overview, the As-Is (especially pain-points and friction-points), the control/regulatory elements and the automated/STP branches, plus existing trends/competitors/ideas. Verify and fill any sourceUrl: pending elements in place. Tools: expandElement(), updateElement()
  2. Step 2 — Scan the webBuild queries from the process title, industry and sub-domain; run several searches and read a promising study in depth. Gather the trends that genuinely bear on this process, not a generic dump. Tools: webSearch(), webFetch()
  3. Step 3 — Draft market trendsDraft each market-trend (blocks The trend/Relevance/Evidence with a specific figure), frontmatter incl. source, sourceUrl, horizon, bearsOn (manual and STP branches, controls, gaps). Give each a tempKey; hold for the Step 5 batch.
  4. Step 4 — Scan competitors (concurrently)Dispatch three read-only sub-agents — EU, global, fintech — in one message; each web-searches named competitors' moves and returns a JSON array of draft specs with type-prefixed tempKeys. Collect the three arrays and hold them. Tools: Task()
  5. Step 5 — Draft ideas, then write the runDerive innovation-idea elements (each addresses at least one real documented problem; fromTrend / fromCompetitor reference temp keys), then write the whole run in one batch and run conformance. Re-work uncovered problems until idea coverage is "complete": true. Tools: createElements(), checkConformance()
  6. Step 6 — ReportMap the per-type counts the batch returned into the canonical report (trends; competitor moves with EU/global/fintech split; ideas), including the conditional web-unavailable and pending-citations blocks only when they apply.
