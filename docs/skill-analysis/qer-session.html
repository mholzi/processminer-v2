qer-session

The conductor for a complete process-documentation session. It runs a fixed six-step frame — select the process, capture its overview, then hand each perspective to its specialist in turn, review across perspectives, validate, and close out — while a resumable QER cursor keeps the long session auditable and restartable.

SME asks to document a process end to end interactive (SME present) WRITES_TO: process overview (root) + delegated to specialists RESUMABLE via QER step cursor · OWNS: the frame, not the content

a What it does

qer-session is the orchestration backbone for documenting a banking process with a subject-matter expert. It is explicitly a conductor: it runs the session frame and dispatches each perspective's work to its specialist skill. It never elicits or writes elements itself — the specialists do that.

The session follows one fixed sequence and never reorders or skips steps:

  1. 1 SELECTIdentify the process and read the cursor.
  2. 2 OVERVIEWCapture the process-level frame with a Y/E/R loop.
  3. 3 PERSPECTIVE PASSESSix cursor positions — one specialist per perspective, in registry order.
  4. 4 CROSS-REVIEWEach specialist reviews the other perspectives' elements.
  5. 5 VALIDATIONDeterministic conformance check plus a judgement gap-analysis pass.
  6. 6 DONEMark the session done and present the canonical close-out.

Two principles govern the whole run. Deterministic frame: the six steps run in order, the cursor is auditable, and the SME — never the skill — decides what to work on and when it is done. You own the frame, specialists own the content: SELECT, OVERVIEW, cross-review, validation and completion belong to qer-session; element elicitation belongs to the dispatched specialist. Everything written in a session is status: draft — the SME approves later, in the web app. Specialists write each element to disk as it is confirmed, so an interrupted session loses nothing.

The skill is careful to distinguish its own cursor from the foundational-run cursor: every getSessionStatus() and advanceSession() call passes kind: "qer", because omitting it would read or advance the foundational cursor by mistake. startSession() is QER-specific and needs no kind.

b The sequence

Each step ends by advancing the QER cursor — that is what moves the session forward and lands a resume in the right place.

  1. SELECT — identify the processIf the invocation names a process, use it. Otherwise greet the SME and either list existing process slugs for them to pick (reading the current overview so the session extends rather than duplicates), or run the new-process skill to scaffold a fresh JSON document.
  2. SELECT — read the cursorRead the QER step cursor. If exists: false or done: true it is a fresh session: start it, use the SME name/role handed over by the session-scope preamble verbatim, advance, and go to OVERVIEW. If exists: true, done: false a session is in flight: tell the SME "Resuming … at step {current}" and jump straight to that step. Tools: getSessionStatus({ slug, kind: "qer" }) startSession({ slug }) advanceSession({ slug, kind: "qer" })
  3. OVERVIEW — draft and confirm the frameCapture purpose, trigger, frequency, volume, scope-in and scope-out. Draft it — never make the SME write prose — then confirm with the verbatim Y/E/R loop, always offering all three choices. On [Y], write the overview by addressing the process's root id (the root meta.id). The overview is the process page, not a collection element. Tools: updateElement({ id, patch }) advanceSession()
  4. PERSPECTIVE PASSES — the dispatch loopSix cursor positions, in registry order: Process → Control & Compliance → Client Journey → Innovation → IT Architecture → Target Process. For each: read status; if skillBuilt is false, tell the SME and advance; otherwise announce the perspective, dispatch its skill in orchestrated mode (read its SKILL.md and run only its perspective phases, skipping its own setup/overview/validation), then advance and checkpoint with the SME ("{Perspective} done — N elements drafted. Continue, or pause?"). Target Process runs last because it synthesises the others. Tools: getSessionStatus() advanceSession()
  5. CROSS-PERSPECTIVE REVIEWOnce two or more perspectives are documented, dispatch each built specialist again for a review-only pass — each reviews the other perspectives' elements from its own lens (a step with no control, a touchpoint with no step behind it, a system nothing references) and surfaces clarifying questions for the SME. With only one perspective documented, state it needs at least two and skip. Tools: advanceSession()
  6. VALIDATIONFirst run the deterministic conformance check, which lists every element whose blocks don't match its schema template. Then do a judgement gap-analysis pass over everything written this session — missing required links, steps with no owner or rationale, orphans, anything thin — and surface each finding (and each conformance issue) as a short clarifying question the SME can answer, defer or skip, applying answers through the owning specialist. Tools: checkConformance({ slug }) advanceSession()
  7. DONE — close outAdvance the cursor a final time, which marks the session done, then present the canonical close-out verbatim, filling only the {process} / {n} / {type} placeholders. Never set approved — that is the SME's call, in the app. Tools: advanceSession()

Biggest win: parallelising the cross-review fan-out (Step 4) on a shared, precomputed document snapshot. Cross-review is the most expensive part of the frame — every built specialist re-reads the whole process to flag inter-perspective gaps — and it is embarrassingly parallel because the document is frozen and the passes only read. Running the specialists concurrently against one cached snapshot turns the slowest serial stretch of the session into a single wall-clock pass, and the shared snapshot makes every reviewer see exactly the same document, removing a source of run-to-run drift at the same time.
