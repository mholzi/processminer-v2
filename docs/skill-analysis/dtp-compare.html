dtp-compare

Review-only sibling of dtp-regenerate: measures a chosen procedure document (DTP) against the corrected As-Is wiki and emits one critical-review finding per material discrepancy — without rewriting the document or producing any new artifact.

Non-interactive · button-invoked WRITES_TO: runtime store (comparison history) READS: As-Is wiki + chosen DTP Never writes the wiki JSON

a What it does

dtp-compare checks an existing DTP — the bank's procedure document — against the corrected As-Is wiki. An analyst has worked the As-Is end to end, so the wiki holds a better, corrected current state than the document. The skill produces exactly one thing: a critical review, the chosen DTP measured against the wiki, one finding per material discrepancy.

It is the lightweight half of the DTP Enhancer. Where dtp-regenerate both rewrites the procedure and reviews it, dtp-compare is review-only: it does not regenerate or rewrite the DTP, and it writes no new document artifact. It is invoked by the DTP Enhancer's "Select a source DTP" action, so it is the right tool when a reviewer wants to check an existing document against the analysis without paying for a full rebuild.

Key constraints and behaviours:

  • Non-interactive. Like area-summary, it reads, reviews and reports — no SME questions, no approval loop. Invoked with a process <slug> and the DTP filename under raw-sources/<slug>/.
  • The wiki is the authority. Where the wiki and the DTP disagree, the wiki wins, and that disagreement is a finding. Approved content is the trusted current state; draft/in-progress elements are weaker ground for a claim.
  • No artifact. Unlike dtp-regenerate, it never writes a regenerated Markdown file — only the comparison findings land, as a new entry in the runtime past-comparison history.
  • As-Is only. One DTP per run, against the As-Is — never the target/to-be.
  • Read-only on the wiki. It never creates, edits or approves wiki elements, never resolves conflicts, and never modifies the process JSON. Everything stated must trace to a wiki As-Is element or the reviewed document.

b The sequence

Four steps run in order: read the document, read the corrected wiki, critically review, then store the review.

  1. Step 1 — Read the DTPRead the document at raw-sources/<slug>/<file> (PDF, Markdown, Word and text read directly). If no filename was given, list the folder and take the most recently ingested document, ignoring any file flagged generated in uploads.json. Study its section structure so it can be walked methodically.
  2. Step 2 — Read the corrected As-Is wikiRead the overview, then every As-Is element — process-step, role, exception, pain-point, metric, process-gap, country-variation, plus control/system where the DTP covered them. Tools: expandElement()
  3. Step 3 — Critically review the DTPWalk the DTP section by section and emit one finding per material discrepancy: { kind, headline, dtpSays, wikiSays, elements, severity, rationale, suggestedDisposition }. kind is outdated / missing / contradiction / added; severity high/medium/low; disposition accepted (the DTP is wrong) or dismissed (likely the wiki is wrong, reconcile it) — a hint only, the reviewer decides.
  4. Step 4 — Store the reviewHand the findings to the writer with the reviewed sourceFile, the findings and coverage.dtpSections (every section walked, even findingless ones). The tool stamps DTPF-… ids, stores the comparison as a new entry in the runtime past-comparison history (no artifact), and returns the run id + count for the one-line report. Tools: writeDtpComparison()

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

IdeaTypeEffortWhy it helps
Deterministic section-to-element alignment. Before review, build a fixed map of each DTP section to the wiki elements that cover it (by step title / keyword overlap), then review one aligned pair at a time.determinismmedTurns Step 3's holistic free comparison into a fixed walk where every section is visited exactly once — exhaustive and reproducible instead of relying on the model to track what it has covered.
Parallelise the per-section comparison. Fan the aligned pairs out across read-only sub-agents, one per section, and merge the findings.speedhighSection comparisons are independent; concurrent fan-out collapses a linear walk over a long DTP into one pass — the single biggest wall-clock lever for review-only.
Strict findings schema, one row per discrepancy. Validate each finding against the seven-field shape with enum-checked kind/severity/disposition before it enters the report.determinismlowCatches a malformed finding pre-write instead of failing in writeDtpComparison, killing retry loops and keeping the comparison history auditable, one row per discrepancy.
Cache the parsed As-Is wiki. Index every As-Is element once into a shared id→element map rather than re-expanding bodies during the walk.speedlowRemoves the repeated expandElement round-trips that dominate Step 2–3 latency.
Chunk large DTPs, map-reduce. Split a long document into section chunks for the comparison map step, then reduce to a deduplicated finding set.speedmedKeeps each comparison inside a tight context window so a long bank procedure isn't skimmed or truncated.

Biggest win: a deterministic section-to-element alignment that the per-section comparisons can then fan out across sub-agents. Because dtp-compare is review-only, almost all of its cost is the holistic walk in Step 3. Aligning each DTP section to its covering wiki elements up front makes the review exhaustive and identical run-to-run; fanning the independent pairs out concurrently then removes the linear latency entirely. A strict one-row-per-discrepancy findings schema on top makes the comparison history fully auditable and retry-free.
