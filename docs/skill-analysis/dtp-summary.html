dtp-summary

Writes an Amazon-style executive memo for one DTP comparison run — a tight narrative for a transformation lead or department head that says, in plain prose, how far the procedure document has drifted from the corrected As-Is and where to focus.

Non-interactive · button-invoked WRITES_TO: runtime store (memo on the run) READS: comparison findings (inline) + overview Never writes the wiki JSON

a What it does

dtp-summary is the leadership-facing capstone of the DTP Enhancer. It takes a single completed comparison run — the gap between an existing procedure document and the corrected As-Is wiki — and writes a short executive memo about it. The reader is a transformation lead or head of department, not an analyst: they want the shape of the problem and where to focus, not a finding-by-finding list.

It does no comparison of its own. The findings from a dtp-compare (or dtp-regenerate) run are handed to it inline in the request — each with a kind, severity, one-line headline, what the DTP says, what the wiki holds, a rationale and the implicated wiki elements — and the skill's job is purely to frame them into a narrative. It is invoked by the DTP Enhancer's "Generate executive summary" button with a process <slug> and the comparison's <runId>.

Key constraints and behaviours:

  • Non-interactive. Like area-summary, it reads, writes the memo and stores it — no SME questions, no approval loop.
  • Works from the findings given. It uses the findings passed inline; for framing it may read the process overview only. It does not read or rewrite the DTP, and does not re-derive the comparison.
  • Memo shape is fixed. A one-paragraph bottom-line verdict (current / drifting / materially out of date), 2–4 themed paragraphs of what's changed (grouped by theme, never by finding id), risk & compliance call-outs, and a prioritised "where to focus" — roughly 200–350 words, specific and quantified where the findings allow.
  • One run per invocation. It summarises a single run; the memo is stored on that run in the runtime store.
  • Read-only on everything else. It never regenerates or edits the DTP, never creates/edits/approves wiki elements, never changes a finding's disposition, and never modifies the process JSON. Everything stated must trace to the findings handed to it or the overview.

b The sequence

Three steps run in order: take in the findings, write the memo, store it on the run.

  1. Step 1 — Read the inputsUse the findings handed in inline — each carries kind (outdated / missing / contradiction / added), severity, a one-line headline, dtpSays, wikiSays, a rationale and the implicated wiki elements. Optionally read the process overview for framing only. Tools: expandElement()
  2. Step 2 — Write the memoWrite a tight Markdown narrative: a one-paragraph bottom line leading with the verdict and its single biggest reason; 2–4 themed paragraphs of what's changed (by theme — automation, ownership, missing controls — not by id); a risk & compliance call-out for anything touching controls, regulation, key-step owners or high-severity gaps; and a short prioritised "where to focus" framed as guidance, never as edits. ~200–350 words, quantified where possible, no preamble.
  3. Step 3 — Store the memoPass the Markdown memo to the writer keyed by slug and run id. It stores the memo on that run in the runtime store — never the wiki JSON — and the skill reports exactly one line: "Executive summary written for {runId}." Tools: writeDtpSummary()

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

Precompute the finding rollup. Cache the overview per slug. Reuse a parsed overview across the compare/summary runs of the same process instead of re-reading it.speedlowThe summary usually runs right after a comparison on the same slug; reusing the overview removes a redundant read.
Biggest win: push the rollup, theme bucketing and risk-set selection into the tool layer and hand them to the skill alongside the inline findings. The memo's whole job is framing — but today the model also has to count the findings, infer the themes and spot the control/owner gaps before it can write a word, and all three drift between runs. Doing that arithmetic and grouping deterministically makes the quantified claims exact, guarantees the risk & compliance section is never missed, and reduces the model's task to the prose itself — faster and far more reproducible, with a fixed four-part template enforcing the shape.
