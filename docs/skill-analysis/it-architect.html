it-architect

Maps the systems landscape a process runs on — the applications and platforms it touches, and the integrations between them — linking each system to the steps it serves and treating every manual re-key the SME describes as a missing integration.

Interactive · Y/E/R loop LENS: the systems landscape — systems of record, hand-offs, data flow OWNS: system · integration Invoked by: free chat / Deep Dive / qer-session

a What it does

it-architect owns the IT Architecture perspective — the two element types system (an application or platform the process runs on) and integration (a connection between two systems, and what flows). It thinks like a solutions architect: where data lives, how it moves, and where a manual re-key betrays an integration that should exist. Unlike the innovation and transformation specialists, it does no web sourcing and consumes no upstream stub — it elicits the landscape directly from the SME on top of the documented process steps.

A defining structural rule: the system↔step link is not stored on the system. It lives on each process-step's systems field, and the system's "Steps" view is derived from it. So linking a system to the steps it serves means patching those steps — a cross-element write the skill performs with updateElement against the process-step, not the system. A system reachable from no step, and a step touching no system, are both findings.

Key constraints and behaviours:

  • Progressive elicitation. Start broad — "name the systems anyone touches" — then drill in, preferring multiple-choice questions primed with banking-IT knowledge (CRM, core banking, screening, workflow, document management) with an "Other — let me describe it" escape hatch.
  • A manual re-key is a missing integration. When the SME describes copying data between two systems by hand, capture both systems and flag the gap for the Process specialist — it is a candidate integration.
  • Narrative-first for integrations. Ask the SME to talk — what data moves, in which direction, how — then extract the structured fields; each integration names exactly the two systems it joins.
  • Stays in its lane. Process steps, exceptions, roles, controls, CX and target-state elements belong to other specialists; mentions are noted and flagged, not authored here.
  • Everything is status: draft with an honest confidence; the SME approves in the app. The skill never sets approved.

b The sequence

Four phases run in order. Under qer-session orchestration the process is already selected and its overview captured, so Phases 0, 1 and 4 are skipped — start at Phase 2. Invoked standalone, run every phase.

  1. Phase 0 — SetupUse the SME name/role from the invocation; only ask if absent. Identify the process, read its overview and the existing elements — especially the process-steps — so systems can be linked to the steps they serve. Tools: expandElement()
  2. Phase 1 — OrientationRead the documented process steps (don't re-document them) and confirm with the SME which steps are system-supported, so you know where each system is used. Tools: expandElement()
  3. Phase 2 — SystemsEvery application or platform the process runs on — its purpose, role and type — drafted and run through Y/E/R. Link each system to the steps it serves by patching each of those process-steps' systems field. Walk the process so none is missed. Tools: createElement() updateElement()
  4. Phase 3 — Integrations[A]/[E]/[N]. The connections between systems — what connects, what data flows, in which direction — captured narrative-first, each linking the two systems it joins. A manual hand-off is a candidate integration; capture the systems and flag the gap. Tools: createElement()
  5. Phase 4 — Validation & close-outSweep for systems linked to no step, steps touching no system, integrations naming fewer than two systems, a re-key described but no integration captured; surface each as a short question, then emit the canonical close-out verbatim with per-type counts. Tools: checkConformance() checkTransitions()

Biggest win: mine the step↔systems links that already exist to auto-derive integration candidates and the coverage matrix. Because the system↔step relation lives on each process-step, the data needed to spot both "two systems meet on one step → likely integration" and "system on no step / step on no system → finding" is already in the document. Surfacing candidate integrations for confirm-or-drop, and the coverage gaps as live facts, converts the two most judgement-light parts of the session — Phase 3 and the Phase 4 sweep — into deterministic, repeatable passes, with a transactional system+step write removing the multi-call linking overhead on top.
