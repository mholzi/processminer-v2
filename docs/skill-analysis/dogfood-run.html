dogfood-run

An end-to-end test harness that creates a throwaway process and drives the entire skill roster through the running app's UI — ingest, foundational run, web sourcing, specialist refinement, comments, lint and a destructive re-ingest — while playing a banking SME at every prompt, then writes a pass/fail QA report and a self-assessment, and even auto-tunes its own walkthrough for the next run.

CLI-only · /dogfood-run Drives the running web app UI end-to-end Plays a banking SME Writes a pass/fail QA report

a What it does

dogfood-run is not a content skill — it is an automated end-to-end test harness for the Processminer web app. In one run (1–3 hours) it creates a brand-new throwaway process and drives it through the entire skill roster, exactly as a real banking subject-matter expert would by clicking through the UI, then writes a pass/fail QA report that the app itself serves on the Mac Mini's LAN IP.

It is CLI-only: it runs only when the user types /dogfood-run in the terminal. It must never be auto-routed from the app's free chat and never run against a real process — the test process is deliberately named "Funds Release Dogfood", distinct from any real funds-release process, so the destructive Stage 8 re-ingest can't touch real content.

The harness wears two hats at once:

  • The test driver — drives the browser via the preview_* MCP tools, observes the app, records evidence, and judges each stage pass or fail.
  • The SME — at every interactive prompt it answers as a knowledgeable but slightly impatient payment-operations expert: mostly approving, deliberately editing a few times, rejecting at least once, and posting real reviewer comments. It never breaks character to the app's chat; the test-driver judgement stays in its reasoning and the report.

Key behaviours and constraints:

  • Two Claude instances kept straight. The harness is the outer Claude (CLI + browser tools). The app's chat spawns its own claude worker that runs the wiki skills. The harness never calls a wiki skill itself — it types messages into the chat and the app's worker runs them.
  • Browser-only via preview_*. Never Bash for browser work, never the Chrome MCP. There is no preview_snapshot — page state is read with preview_eval DOM queries; preview_screenshot is for live verification only and cannot be written to disk, so all report evidence is textual.
  • SME answers are grounded in the fixtures. Two paste-text source documents drive the run: fixtures/source-doc-v1.md (the initial ingest) and fixtures/source-doc-v2.md (deliberately contradictory — changed STP SLA, cut-off, Treasury threshold and a control owner) for the Stage 8 conflict edge case. The SME must not invent facts that contradict them.
  • Resumable. After every stage it writes state.json under public/test-report-assets/<run-id>/; /dogfood-run <run-id> reloads it and skips completed stages. No run's process is ever deleted.
  • The loop closes on itself. Stage 9 gets an independent SME verdict on the produced content; Stage 10 traces every shortfall to a [walkthrough] gap (fixable here) or a [skill] finding, then auto-edits this very SKILL.md — additively, never touching the scope-locked sections — and logs each edit to REVISIONS.md.

b The sequence

Twelve checkpointed stages. Each is asserted PASS/FAIL with a one-line reason, a textual DOM snapshot is captured as evidence, state.json is written, and the run continues even past a FAIL. There are no wiki-skill tool calls here — the harness drives the UI and the app's worker runs the skills.

  1. Stage 0 — PreflightStart the dev server bound to 0.0.0.0 so the report is LAN-reachable; read the LAN IP (ipconfig getifaddr en0); form the run id YYYY-MM-DD-HHMM; create the run folder. Open the app, confirm it loads, the process switcher is present and the console is clean. Tools: preview_start(), preview_eval()
  2. Stage 1 — Create processOpen the switcher's "new process" action; the app runs new-process. As the SME give the name "Funds Release Dogfood" and confirm description/slug/abbreviation with [Y]. Assert: the process appears in the switcher and wiki/processes/<slug>.json exists with root meta + overview. Tools: preview_click(), preview_fill()
  3. Stage 2 — Upload and ingestOpen "⬆ Upload document", switch to "Paste text", name it funds-release-dogfood-v1.md, paste the full fixtures/source-doc-v1.md, click "Add document". The app saves to raw-sources/<slug>/ and runs document-ingest (20+ min). Assert: triage screen appears; report stored in the ingest field; a dozen-plus elements created across the typed arrays. Tools: preview_fill(), preview_eval()
  4. Stage 3 — Foundational runFrom triage, start the foundational run and walk every challenged element. Approve most; make at least one [E] edit with a concrete correction and at least one [R] reject forcing a redraft. Assert: the cursor (in data/runtime/<slug>.json reviewState) reaches completion; approved elements are stamped; the edit and redraft are visible. Tools: preview_fill(), preview_console_logs()
  5. Stage 4 — Source from the webTrigger each "✦ Source from the web" empty-state CTA: source-regulation, source-cx, source-innovation, and the Target Process source-target. These do live web research. Assert: each produces drafts; sourced elements carry a relevant/disregarded relevance state (not approval); source-target yields target-state, transformation-decision and gap drafts. Tools: preview_click(), preview_eval()
  6. Stage 5 — Specialist refinementUse Deep Dive on 2–3 elements and triage some sourced elements relevant/disregarded, then run eight specialist sub-passes (each front-loaded with the full SME brief so the specialist runs to one commit-cycle): 5a client-journey (internal CX, ≥4 friction-points), 5b transformation-agent (to-be per As-Is step, gap-resolution per gap), 5c control-compliance (≥3 audit-findings, ≥5 exceptions, control per step), 5d process-specialist (process-gaps), 5e innovation-analyst (prioritisation + risk per idea), 5f add-entry on Metrics, 5g it-architect (integrations + system attributes), 5h add-entry on Country-Variations. Assert: each sub-pass populated its section or left an explicit "accepted as empty" note. Tools: preview_click(), preview_fill()
  7. Stage 6 — Comments and comment-reviewPost real reviewer comments (challenges, missing-link questions) on 2–3 elements, then click "✦ Review with analyst" on one to run comment-review. Assert: comments appear in notes; the analyst resolves them with a closing summary; an element whose content the review changed reverts approved → in-progress. Tools: preview_fill(), preview_eval()
  8. Stage 7 — LintClick "⊛ Run lint" and let run-lint finish. Assert: the lint report is written to data/runtime/<slug>.json (lint, not a sidecar) with findings; the Review panel shows them; any implicated approved element reverts to in-progress stamped run-lint. Tools: preview_click(), preview_eval()
  9. Stage 8 — Conflict edge caseRe-upload fixtures/source-doc-v2.md (contradicts v1: STP SLA 2→4h, cut-off 14:00→12:00 CET, new callback step, Treasury threshold EUR 5m→2m, C-3 owner change). Let document-ingest flag conflicts, then run conflict-resolution and walk each — accept some, keep some, edit one. Then a stale-reference sweep: ask the assistant to grep the wiki for superseded values and patch every remaining occurrence. Assert: conflicts flagged and cleared per the SME's decisions; ≥1 approved element reverted to in-progress. Tools: preview_fill(), preview_eval()
  10. Stage 9 — Process SME assessmentLaunch a subagent (general-purpose) with a fully self-contained prompt, briefed as a senior payment-operations SME. It reads the whole process JSON + runtime store, scores empty/sparse and populated sections against the schema, gives per-area 0–10 scores, lists the top three gaps, and verdicts auditor-readiness in ≤800 words. Assert: the subagent ran and returned an assessment (FAIL only if it couldn't read the wiki). Tools: Agent()
  11. Stage 10 — Walkthrough tuningTrace every Stage 9 shortfall to a [walkthrough] gap (fixable here) or a [skill] finding (not). Write the prioritised list to walkthrough-tweaks.md, then auto-apply the [walkthrough] tweaks by editing this SKILL.md — additive only, never the scope-locked sections — logging each to REVISIONS.md and re-checking the file is still well-formed. Assert: every shortfall is accounted for and the SKILL.md is valid. Tools: Edit()
  12. Stage 11 — ReportAssemble the self-contained QA report (verdict header, stage table, per-stage detail, the SME assessment with a section inventory, the walkthrough-tuning split, the process artifact, and a consolidated errors/findings list) and write it to public/test-report.html and the archived public/test-report-<run-id>.html. Tell the user the LAN report URL. Tools: Write()

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

IdeaTypeEffortWhy it helps
Scripted SME persona answers from fixtures. Ship a fixtures/sme-script.json keyed by stage/element with the exact [Y]/[E]/[R] decision and edit text, instead of improvising each reply.bothmedThe single biggest source of run-to-run variance is the SME improvising different corrections each time. A fixed script makes the produced content reproducible and removes per-prompt reasoning latency.
Fixed assertion checklist per stage. Encode each stage's assertions as a static checklist (selector + expected DOM/JSON predicate) the harness evaluates, rather than free-form judgement.determinismmedToday "did this stage pass?" is a per-run model judgement that can drift; a declared predicate set makes pass/fail repeatable and the health score comparable across runs.
Snapshot + diff the process JSON between stages. Save wiki/processes/<slug>.json after each stage and assert the expected delta (N new steps, cursor advanced, owner changed) by diff.determinismmedReplaces "read the DOM and eyeball it" with an exact structural assertion — the same input always produces the same pass/fail, and regressions surface as concrete field diffs.
Pre-seed fixtures to skip live web sourcing. Add a test mode where Stage 4's source-* CTAs draft from canned fixture results instead of live web research.bothhighWeb sourcing is slow and non-deterministic (results change daily). Canned fixtures cut the longest non-deterministic stage to seconds and make the resulting drafts identical every run.
Drive at the API level, not UI clicks. Where a stage just needs an action triggered (upload, run-lint, source CTA), POST to the existing endpoints instead of preview_fill/preview_click.speedhighDOM polling for the Working…→Message the assistant… placeholder flip is brittle and slow; calling the session/findings/upload endpoints directly removes click latency and selector fragility while still exercising the real backend.
Parallelise independent stage checks. Run the Stage 5 specialist sub-passes that touch disjoint sections (e.g. 5e innovation, 5g IT, 5h country) concurrently rather than serially.speedhighStage 5 is the wall-clock bulk (eight slow sub-passes, ~15–25 min/turn as the JSON grows). Sub-passes on non-overlapping element types have no ordering dependency and can overlap.
Cache the login/session. Persist the authenticated cookie/session across resumes and stages so each preview turn isn't re-authenticating.speedlowRemoves repeated bootstrap-admin login round-trips, especially on resume where the run reconnects to a fresh browser context.
Structured QA-report schema. Have the harness emit a machine-readable report.json (stages, assertions, scores, findings) and render the HTML from it deterministically.determinismmedDecouples the verdict data from prose, lets runs be diffed/trended over time, and guarantees the HTML can't disagree with state.json.
Fail-fast gating on preconditions. Hard-stop a stage if its precondition assertion fails (e.g. Stage 3 won't start if Stage 2 created zero elements) instead of pressing on into a long broken stage.speedlowAvoids burning an hour of downstream stages on a wiki that the ingest never populated; the failure is caught at its origin with a clear cause.
Seedable randomness for SME variety. When the SME does vary (which element to [R], which to comment on), drive the choice from a logged seed in state.json.determinismlowKeeps the desirable coverage variety while making any individual run exactly replayable for debugging a flaky stage.
Biggest win: a scripted SME persona (from a fixture) plus snapshot-and-diff of the process JSON between stages. Today the harness's two slowest, least repeatable parts are the SME improvising fresh answers at every prompt and the test driver eyeballing the DOM to judge each stage — so two runs of the "same" test produce different content and different verdicts. Pairing a fixed answer script with structural JSON diffs turns the run into a deterministic regression test: identical inputs yield identical wiki output and an identical pass/fail, which is exactly what a dogfood harness needs to catch real regressions instead of attributing every difference to model variance. Layer the API-level drive and pre-seeded sourcing fixtures on top and the 1–3 hour run collapses toward minutes.
