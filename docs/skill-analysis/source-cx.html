source-cx

Autonomously sources the external client-experience layer from the web — how European, global and fintech competitors run the same client journey, plus industry CX benchmarks — and drafts competitor-cx-* and cx-benchmark elements. It never touches the bank's own journey.

Non-interactive · button-invoked SOURCE: web · provenance=web FILLS: Competitor CX · CX Benchmarks Triaged, not approved

a What it does

source-cx is the CX counterpart of source-innovation. Invoked with a process <slug>, it sources only the comparative client-experience layer: how competitors run the same client journey, and what the industry treats as the CX standard. It writes competitor-cx-eu, competitor-cx-global and competitor-cx-fintech into competitor-cx, and cx-benchmark into cx-benchmarks.

It pointedly does not document this bank's own journey — its channels, touchpoints, moments and friction are captured by the SME or document-ingest, not the web. The skill closes its report with a verbatim handoff line making that boundary explicit, so a user who clicks "Source from the web" on the Client Experience area is not misled into thinking the whole area was filled.

Key constraints and behaviours:

  • Non-interactive. No SME, no Y/E/R, no approval — everything is status: draft for later triage in the app.
  • Three concurrent competitor tiers. EU / global / fintech are independent research streams, dispatched as three read-only sub-agents in a single message; each returns only a JSON array of draft specs.
  • Real competitors only. Named real peers and real cited sources — a handful of material examples per tier, never a dump, never invented.
  • Provenance = web. Each block (The journey / Relevance / Evidence for competitor CX; The benchmark / Relevance / Evidence for benchmarks) carries source: web with URL + snippet + fetch date, blocking approval until a specialist confirms with the SME.
  • Extend, never duplicate. It reads existing competitor-cx-* and cx-benchmark elements first.

b The sequence

Four prescribed steps; competitor tiers are written incrementally as each returns.

  1. Step 1 — Read the processRead the overview and the documented client journey (channels, touchpoints, moments, friction-points) to know what journey to benchmark, plus existing competitor-cx-* and cx-benchmark elements so it extends rather than duplicates. Tools: expandElement()
  2. Step 2 — Scan competitor CX (concurrently)In one message, dispatch three read-only sub-agents — EU, global, fintech — each briefed to web-search named competitors' client experience (onboarding, channels, speed, self-service, reviews) and return a JSON array of draft specs (blocks The journey/Relevance/Evidence; competitor, source, sourceUrl; a web provenance map). Write each tier as soon as it returns, keeping a running per-type tally. Tools: Task(), createElements()
  3. Step 3 — Scan CX benchmarksWeb-search industry CX benchmarks and client-expectation research (onboarding-time standards, NPS/effort, what corporate clients expect) and write each cx-benchmark as its own createElements group. Then run conformance once over the run. Tools: createElements(), checkConformance()
  4. Step 4 — ReportMap the running per-type totals into the canonical report (Competitor CX total with EU/global/fintech split; CX benchmarks; sources), then append the verbatim handoff line that the internal journey was not touched and needs the client-journey-specialist. If web search was unavailable, add the fixed line saying so.
