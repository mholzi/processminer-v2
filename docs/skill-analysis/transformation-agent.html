transformation-agent

The forward synthesis — turns everything the five other perspectives documented into a coherent Target Process: the target state, the decisions to reach it, the requirements that operationalise them, the boundary dependencies, and the gaps to close — with coverage traced back to every open As-Is problem.

Interactive · Y/E/R loop LENS: forward synthesis — As-Is + innovation → a workable to-be plan OWNS: target-state · transformation-decision · requirement · process-dependency · gap · assumption Invoked by: free chat / Deep Dive / qer-session

a What it does

transformation-agent owns the Target Process perspective — the downstream synthesis of every other lens. It does not invent a future state; it reads the documented As-Is (pain-point, process-gap, process-step), the risk picture (compliance-gap, audit-finding), the friction points, the innovation work (innovation-idea, innovation-risk) and the systems landscape, and turns them into a plan. The Target Process is usually stubbed first by the non-interactive source-target skill, and this skill refines that stub with the SME and adds what the consolidation missed.

Its spine is traceability and coverage. A target-state replaces the As-Is steps it touches (driving the As-Is↔To-Be overlay); a transformation-decision resolves the problems it fixes and realises the target states it brings about; a requirement must carry derivedFrom; a gap links the target it serves. Every open As-Is problem should be resolved by at least one decision — an uncovered problem is either a real gap or a missing decision, never silently dropped.

Key constraints and behaviours:

  • No web research. The innovation behind the target is already in the wiki (source-innovation / innovation-analyst); this skill synthesises from documented work only.
  • Preserve stubbed frontmatter on edit. On [E], patch only the corrected block via updateElement so a decision's resolves / realises, a target-state's replaces and a gap's target link survive untouched.
  • Requirements are testable and sourced. Each carries reqType (FUNCTIONAL / NON-FUNCTIONAL), a moscow priority, and — required — derivedFrom. A requirement with no traceable source is not a requirement and is never written.
  • Assumptions captured as they surface. Whenever the target rests on something unconfirmed, write an assumption with its assumptionStatus and required bearsOn — not a phase of its own.
  • Stays in its lane. It reads every perspective freely but never authors As-Is or innovation elements; it points the SME at council-review to challenge the target before approval.

b The sequence

The phases run in order; assumptions are captured the moment they surface, not in a phase of their own. Under qer-session orchestration, Phase 0 and Phase 5 are skipped, but Phase 1 always runs.

  1. Phase 0 — SetupUse the SME name/role from the invocation; only ask if absent. Identify the process and read its overview. Tools: expandElement()
  2. Phase 1 — OrientationRead the perspectives the target builds on — pain-points, process-gaps, steps, compliance-gaps, audit-findings, friction-points, innovation ideas and risks, the systems — then the existing source-target stub. Confirm which problems matter most and which ideas the bank will actually fund. Tools: expandElement()
  3. Phase 2 — Refine target statesWalk each target-state one at a time with Y/E/R; on [E] patch only the corrected field to keep replaces intact. Add missing target states via [A]/[E]/[N], capturing replaces on each. Tools: updateElement() createElement()
  4. Phase 3 — Transformation decisions[A]/[E]/[N]. Each decision captures resolves (the open As-Is problems it fixes) and realises (the target states it brings about) — these drive the computed coverage check. Tools: createElement()
  5. Phase 3b — Requirements[A]/[E]/[N]. Operationalise decisions into testable requirement elements with reqType, moscow and required derivedFrom; optionally addresses a gap. Tools: createElement()
  6. Phase 3c — Dependencies[A]/[E]/[N]. The target boundary — process-dependency elements with direction, atStep, what crosses and why. The to-be boundary, not the As-Is one. Tools: createElement()
  7. Phase 4 — Gaps[A]/[E]/[N]. Any open problem no decision resolves, any target state no decision realises, any innovation-risk the transformation must manage is a candidate gap; each links the target-state it serves. Tools: createElement()
  8. Phase 5 — Validation & close-outSweep for target states with no innovation behind them, decisions resolving nothing, open problems uncovered, gaps untraced; surface each as a short question, emit the canonical close-out with per-type counts, and point the SME at council-review. Tools: checkConformance()

Biggest win: make coverage a computed artifact the skill consumes rather than a sweep it re-reasons at the end. With a live open-problem→decision coverage map and a precomputed upstream index, the model wires resolves / replaces / derivedFrom by id as it drafts, and Phase 4 collapses to confirming auto-derived gap stubs for whatever the graph shows uncovered. That single change makes the skill's core invariant — every open As-Is problem covered by a decision or a named gap — deterministic and identical run-to-run, while cutting the longest, most error-prone stretch of the session.
