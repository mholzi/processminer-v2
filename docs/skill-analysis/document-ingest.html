document-ingest

Reads an uploaded source document and extracts its content into the wiki as draft elements — fanning out drafter and verifier sub-agents per element-type group, verifying every claim against the source, filling the overview, and flagging conflicts. No SME is in the loop; the document is the only authority.

Invoked by: "⬆ Upload document" modal after the file is saved Semi-interactive (single Y/N gate, then non-interactive) WRITES_TO: process JSON elements + overview + ingest report OWNS: no fixed type — cross-perspective by nature

a What it does

document-ingest takes a file already uploaded under raw-sources/<slug>/ (Karpathy LLM-Wiki layer 1) and extracts its process content into typed wiki elements. It is the second step in the documentation flow, between new-process and the triage screen / foundational-run. It is cross-perspective: it writes whatever element types the document covers (steps, roles, systems, controls, regulations, metrics, …), all as status: draft. It never sets approved and never overwrites on a conflict.

Its defining constraint is that no SME is present. The document is the sole authority, and an extraction can quietly claim more than the source says — so every drafted element is verified against its source passage before it is written. This is the Verify half of the Brainstorm/Author/Verify pattern, realised "against a source" rather than interactively.

Mechanically it is a map-reduce. A whole document of drafts is too much linear writing for one turn (the legacy linear flow emitted ~315k tokens and triggered mid-flow auto-compact), so the main agent plans an outline of every candidate element, then fans out one drafter sub-agent per element-type group, then one verifier per group, then merges and writes everything in a single batch. The main-agent output drops to ~30k tokens.

Key behaviours and constraints:

  • Verbatim user-facing wording. The summary prompt, the [Y]/[N] gate, and the final extraction report are reproduced exactly from the prompt — never written from memory.
  • tempKey discipline. New elements get deterministic tempKeys (role-1, sys-3); cross-group relations reference sisters by @<tempKey>, which createElements resolves at write time. tempKeys may appear only in relation/field arrays, never in prose blocks.
  • Three outline dispositions. New topic (tempKey), refine (existing id, no contradiction), or conflict (existing id, contradicts) — the last is never drafted, only recorded.
  • Provenance is earned by the verifier. Drafters tag headings provisionally and omit evidence; verifiers trace each claim, strip hallucinations, fill the verbatim evidence quote, set confidence, and demote untraceable headings to proposed.
  • Conflicts are reported, not resolved — they flow to the ingest report and are later worked by conflict-resolution.

b The sequence

One Y/N gate, then a deterministic map-reduce; Step 3 is the engine.

  1. Step 1 — Review and summariseRead the document at the given path; take the slug from the path and the title from the process overview. Write a summary and present the canonical "Shall I extract…?" prompt with the [Y]/[N] choice.
  2. Step 2 — Wait for the userOn [N] stop with the exact "document stays in raw-sources" reply. On [Y] continue to Step 3.
  3. Step 3.1 — Identify the sourceConfirm which uploaded file to ingest (already in uploads.json; no register step). Carry its filename through to the report.
  4. Step 3.2 — Outline every element up frontRead schema/process-schema.json; list existing ids+titles per type. Tools: expandElement({ type }). Read the document end-to-end and emit one outline entry per candidate (new / refine / conflict), partitioned one group per type (split >~10 into -a/-b). Write /tmp/<slug>-outline.json.
  5. Step 3.3 — Draft (fan out, one sub-agent per group)Dispatch all drafters in a single message; wait for all. Each reads its group, reads the schema and (if refining) existing bodies via expandElement({ type, id }), drafts createElement specs / refine specs / conflict entries, omits evidence, tags provenance provisionally, and writes /tmp/<slug>-drafts-<group>.json. Drafters are read-only on the wiki.
  6. Step 3.4 — Verify (fan out, 1:1 with drafters)Dispatch all verifiers in one message; wait for all. Each traces every claim to the source, removes/rewrites unsupported claims, sets confidence, fills provenance.evidence (or demotes to proposed), and writes /tmp/<slug>-verified-<group>.json. Read-only on the wiki.
  7. Step 3.5 — Merge + write (one batch)Concatenate all verified specs and the conflict/correction lists. Tools: createElements({ elements }) — it assigns ids, resolves every @<tempKey>, writes all elements, and returns created + per-type counts.
  8. Step 3.6 — Fill the process overviewDraft the overview body fields from what the document explicitly states (purpose, owner, trigger, frequency, scope in/out, input, output); leave un-stated fields blank; never overwrite a contradicted field (record a conflict) — except description, which is corrected if stale. Verify against source, then Tools: updateElement({ id, patch }) on the root overview id.
  9. Step 3.7 — Check conformanceTools: checkConformance({ slug }). Fix any flagged element; re-verify every document heading's evidence is a verbatim, traceable quote.
  10. Step 3.8 — Write the ingest reportAssemble { file, conflicts, corrections } (plus any overview-level entry). Tools: writeIngestReport({ slug, report }) — writes the ingest field the triage screen reads.
  11. Step 4 — Summarise the extractionEmit the canonical report (created / updated / overview / verified-corrected / conflicts, with one bullet per correction and per conflict), using counts from the tools, not memory.

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

IdeaTypeEffortWhy it helps
Merge drafter+verifier per group into one sub-agent. Have each group sub-agent draft and self-verify against the source in a single pass, writing the verified file directly.bothmedHalves the fan-out rounds (two waves → one) and the per-group file I/O; the source passage is already in that sub-agent's context, so re-reading it for verification is wasted work.
Cache the document map once and pass it to sub-agents. Read the schema + existing-element index in the parent and hand it down, instead of every drafter re-running expandElement.speedlowRemoves N redundant schema/collection reads (one per group) that all return the same data.
Structured-output the outline + draft specs. Constrain outline and draft specs to the schema with strict JSON output rather than free-form /tmp JSON.determinismmedEliminates malformed-spec retries and tempKey-format drift before they reach createElements, and makes the outline machine-validatable.
Skip the verifier for pure-quote headings. When a drafter marks a heading document with an exact span offset, validate the quote deterministically (substring match) instead of an LLM re-trace.bothmedThe most common verification — "does this quote literally appear?" — becomes a string compare, reserving model verification for the inference-across-passages cases that actually need judgement.
Fold conformance into the batch write. Have createElements run checkConformance internally and return failures, instead of a separate Step 3.7 round.speedmedCollapses write + validate into one tool call; failures come back attributed to the spec, avoiding a re-read-and-fix cycle.
Deterministic evidence-traceability gate. Make checkConformance assert every document heading's evidence is a verbatim substring of a raw-sources/<slug>/ file.determinismmedTurns "re-read the passage and confirm" (a judgement step that can pass a bad quote) into a hard, repeatable check — exactly the hallucination class the skill exists to catch.
Generate the final report from tool counts. Render the Step 4 report directly from createElements + writeIngestReport return values.determinismlowRemoves any chance of recounting drift in the user-facing summary and guarantees the verbatim template.
Dynamic group balancing from outline counts. Split/merge groups by actual entry counts (not just the fixed ~10 heuristic) before dispatch.speedlowBalances sub-agent load so the slowest group doesn't gate the whole fan-out wall-clock.
Biggest win: collapse the two fan-out waves into one draft-and-self-verify sub-agent per group, and make quote traceability a deterministic substring check inside conformance. Today the source passage is read twice (drafter, then verifier) across two dispatch barriers, and the integrity guarantee rests on an LLM re-reading the document — slow and fallible. A single per-group pass with a string-match evidence gate cuts the wall-clock roughly in half and replaces the highest-stakes judgement step (did this document quote actually come from the source?) with a check that can't drift between runs.
