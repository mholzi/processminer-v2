council-review

The target-state council: the five perspective specialists who did not author the target each challenge the proposed To-Be from their own lens, and the skill writes their concerns to the process's targetReview field for the SME to triage. Non-interactive: no SME questions, no approval loop.

Non-interactive (review, write, close out) WRITES_TO: targetReview field via writeTargetReview() Invoked by: a button / free chat SCOPE: whole process (target area)

a What it does

council-review runs the cross-perspective review of the transformation — the lint "council" (SKILLS.md §9) pointed specifically at the Target Process area. The target itself is the transformation-agent's work; the council is every other specialist sense-checking it: Process, Control & Compliance, Client Journey, Innovation, and IT Architect. Each puts on its lens and challenges the proposed target, and the skill writes the feedback to the targetReview field for the SME to accept or reject in the app's Council Review panel.

It reviews the transformation-decision and target-state elements and their relations (a decision's resolves/realises, a target-state's replaces), reading the As-Is freely as context. Each feedback item is a genuine concern grounded in a specific decision or target-state element, captured as { specialist, title, detail, targets } — the targets being the ids that re-open on accept. The skill routes by potential relevance, not stored relations: the Control specialist reviews every decision, even those naming no control.

The scope is either the full council (all five) or one named specialist. It is strictly non-interactive — no questions, no approval loop. The writeTargetReview tool id-stamps the items, stamps each triage: pending, and writes the field; the skill never sets a triage state. Each specialist stays in its lane (one lens per item), raises concerns not praise, and an honest empty result writes an empty items list rather than manufacturing findings.

b The sequence

Orient, review specialist by specialist, write, close out — four phases, no SME loop.

  1. Phase 0 — OrientationIdentify the process from the slug, read the schema, then the Target Process area (transformation-decisions, to-be-design) and the As-Is. Note which specialists are in scope (full council or one named). Tools: expandElement({ type }) · expandElement({ type, id })
  2. Phase 1 — Review, specialist by specialistFor each in-scope specialist, put on its lens and challenge the target — a control the change weakens, a To-Be step that is not operable, a touchpoint the redesign breaks, a system capability a target-state assumes but lacks. Ground every item in a specific decision/target-state id; route by potential relevance; raise concerns only, never praise.
  3. Phase 2 — WriteAssemble one object — { ran: [specialists], items: [{ specialist, title, detail, targets }] } — and pass it to the writer, which id-stamps items (R-001…), stamps each triage: pending, and writes the field. Tools: writeTargetReview({ slug, reviewData })
  4. Phase 3 — Close outSummarise how many items each specialist raised and tell the SME to triage them in the Council Review panel (Validation section) where accept re-opens the implicated decision. If nobody raised a concern, still write an empty items list and say the council found nothing to flag.
