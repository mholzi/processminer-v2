Current-state walk

foundational-run

The first guided pass after a document is ingested: a meticulous process analyst reads the whole wiki, then challenges every current-state element in foundational order to tease out rework — and deepens past the document to surface the pain points and missing controls a source never states, approving each with the SME. A stopped run resumes exactly where it left off.

After document-ingest, or SME asks to start/resume the run interactive (SME present, one element per exchange) WRITES_TO: current-state elements + mid-run pain-points / gaps / controls RESUMABLE via foundational element cursor · OWNS: the challenged walk

a What it does

foundational-run walks a freshly-ingested process end to end as a detailed, meticulous process analyst — the persona that misses nothing, reads every line, and asks the one question that proves a "complete-sounding" draft isn't. It is explicitly not an approval queue: it is a deepening pass. The model does the judgement — read, challenge, redraft — and the native tools do the mechanical work: the cursor, approvals and element writes.

"Current-state" is broader than the As-Is Process nav area: it spans As-Is Process, Risk & Compliance, Client Experience and IT Architecture — controls, systems and the rest are all in scope, each challenged through its owning specialist's lens. Forward-looking elements (market trends, innovation ideas/risks, requirements, target state, transformation decisions, gaps) are out of scope.

The defining move is deepening past the document. A source document records what exists; an owning-lens challenge tests only that. Two things slip through both, and document-ingest can reach neither: the lived layer (where the work hurts) and what is absent (a step that should carry a control but has none). On each process-step the analyst catches both — a focused pain-point probe and a control-coverage check — so those sections don't stay permanently empty.

Resumability is structural: a freshly-ingested process has no conflicts (conflicts are a separate flow), and a stopped run picks up from the cursor's reported position. The run is built around a strict provenance contract — ingested elements arrive with interpretive headings marked proposed, and setApproval() blocks approved while any heading is still proposed, so a [Y] is only reachable once every heading has been read back and confirmed.

b The sequence

Read the whole process once, then walk the cursor one element per exchange — never batch the challenge, because the challenge is the value.

  1. Step 1 — Read the whole processOn invocation, read the overview (root meta/content) and every current-state element — list each collection, then read the bodies — and build the complete picture: the spine of steps, every relation, what is thin, what is missing, what does not connect. Give the SME a one-line orientation ("I've read {process} — {n} steps, {n} controls, {n} exceptions … Starting the foundational run"). Tools: expandElement({ type }) expandElement({ type, id })
  2. Step 2 — Build or resume the queueRead the session status. Error / no state → fresh run: build the queue (overview first, then current-state elements in foundational order — process steps, roles, then the rest), and it reports the first item. State exists, not done → resumed run: tell the SME "Resuming … item {position} of {total}" and continue from current. State exists, done → go to Step 4. The output carries position, total, done, current, plus the SME-facing outcomes_line and closeout_template — relay both verbatim, never retyped. Tools: getSessionStatus({ slug }) buildQueue({ slug })
  3. Step 3.1 — Present itShow the SME the current element — the overview, or the element's blocks — and what it links to. The app opens that section in the canvas automatically; the model does not navigate it.
  4. Step 3.2 — Challenge it (owning-lens)Pose 1–3 sharp, specific questions designed to expose weakness — a missing detail, an unsupported assumption, an edge case, an unjustified "why", a number that looks off — through the owning specialist's lens (Process / Control & Compliance / Client Journey / IT Architect / the overview's purpose-trigger-scope frame). Use the whole-process picture from Step 1 to name related elements. Read each proposed heading back to the SME so they confirm or correct it — this is what unblocks a later [Y].
  5. Step 3.3 — Deepen past the documentOn each process-step, in the same message as the challenge: a pain-point probe (is there staff/process pain — a workaround, a recurring delay, manual re-keying, a stalling hand-off? create a pain-point only if named; never invent one) and a control-coverage check (if the step has no control linked, ask whether it is an accepted gap or a control should exist; on a confirmed gap create a compliance-gap and either draft a paired control stub or record the accepted-risk choice in the gap's Remediation block). This beat is elicitation, not challenge — mark only what the SME confirmed as elicited. Other element types get no deepening beat. Tools: createElement({ slug, element })
  6. Step 3.4 — Offer the outcomesPresent the outcomes_line character-for-character on its own line — four outcomes: [Y] accept · [E] edit · [D] deep dive · Move on. [Y]: first reconcile provenance — rewrite each confirmed proposed heading to elicited with the SME's quote (else the approve call fails), then approve under the SME's name. [E]: redraft, write the fix (single block/field patch, or a same-id multi-block redraft), keep frontmatter relation lists in sync with prose, echo one line of what changed, approve, and continue straight to Advance in the same turn. [D]: read the owning specialist's SKILL.md and run a full elicitation here, then approve. Move on: leave it in-progress, set no approval. Tools: updateElement({ slug, id, patch }) updateElement({ slug, element }) setApproval({ slug, id, status, approver })
  7. Step 3.5 — AdvanceRuns after every outcome — [Y], [E], [D] and Move on — in the same turn as the write/approval. Never end the turn after an edit or approval and wait for the SME to say "continue": advancing and presenting the next item is part of the same response. If it reports done, go to Step 4; otherwise present the next current item. Mid-run created elements are not in the queue — they are never challenged this run, stay in-progress, and are tracked in a running list split into current-state vs gaps. Tools: advanceSession({ slug }) checkConformance({ slug, elementId })
  8. Step 4 — Close-outWhen the cursor is done, status reports done: true and a closeout_template. Count approved vs deferred (in-progress) current-state elements, then present the template exactly — fill every {…}, keep each {if …} block only when its condition holds: name mid-run current-state elements explicitly, name mid-run gaps, and list each still-empty section with the skill or ✦ Source-from-the-web button that fills it (never the Target Process sections). Tools: getSessionStatus({ slug })

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

IdeaTypeEffortWhy it helps
Precompute the document map once at Step 1. Build the spine, relation graph, per-step control-linkage map and "what's thin" flags in one tool call, then carry that snapshot through the whole walk.bothmediumStep 1 already requires reading every element; capturing the result as a structured map removes per-item re-reads and makes the control-coverage check (Step 3.3) a lookup instead of a fresh judgement each time.
Deterministic control-coverage flag. Compute "this step has no control linked" from the relation graph and surface it as a boolean on each queue item, rather than the model re-deriving coverage per step.determinismlowSKILLS.md calls a step missing a control the most valuable finding; making it a counted fact means the probe fires on exactly the right steps every run.
Persist the run cursor deterministically. Ensure the element cursor advances and checkpoints atomically with each write/approval, so a crash mid-turn never leaves position and on-disk state disagreeing.determinismmediumThe whole resume guarantee ("item {position} of {total}") depends on the cursor matching disk; atomic advance removes the only window where a resume could land on the wrong item.
Server-rendered outcomes_line & close-out. Both strings are already declared single-source-of-truth and relayed verbatim — render the filled versions (counts, by-type, {if} blocks) server-side and have the model only relay them.determinismlowEliminates the recurring failure mode of the model re-lettering options, dropping [D], or miscounting at close-out — the fixed wording can no longer drift.
Auto-reconcile confirmed provenance on [Y]. When the SME confirms a proposed heading during the challenge, flip it to elicited with the captured quote as part of the approve call, instead of a separate rewrite-then-approve.bothmediumRemoves the most common [Y] failure — approving before reconciling provenance, which the tool rejects — and collapses two writes into one.
Frontmatter-sync on edit. When an [E] rework names a SYS-*/ROLE-*/CTRL-* id in prose, auto-add it to the matching relation list in the same write rather than relying on a follow-up patch.determinismmediumThe skill already warns prose/frontmatter drift is what lint later flags; deriving the relation list from the prose ids prevents the drift instead of curing it.
Batch the process-gap tail. The skill already permits presenting the process-gap tail together — make that an explicit deterministic batch the queue emits, so gaps are reviewed in one labelled exchange.speedlowGaps are cross-referential and low-challenge; batching the known tail removes several one-per-exchange round-trips at the end of the run.
Single-turn challenge+probe enforcement. Validate that the pain-point probe ships in the same message as the challenge (Step 3.3 requires it), e.g. via a structured per-step prompt template.speedlowGuarantees each step costs one exchange not two, exactly as the skill intends, halving the dominant cost of the walk.
Cache expandElement bodies within the run. Key a per-run cache on element id + revision so re-presenting or deep-diving an element doesn't re-fetch an unchanged body.speedlowStep 1 loads everything and Step 3 re-presents many of the same elements; caching avoids redundant reads across the walk.
Precompute the "still to document" list. Derive empty-section → filling-skill mappings (§4/§11 of SKILLS.md) from the document at close-out instead of the model re-checking each section.determinismlowMakes the close-out's third {if} block a deterministic projection — every empty section, mapped to its exact skill or ✦ button, with no section forgotten.
Biggest win: moving the run's fixed wording and gating onto the backend — a server-rendered outcomes_line and closeout_template, an atomically-persisted cursor, a deterministic control-coverage flag, and auto-reconciled provenance on [Y]. These are the four places the skill is most fragile: the model re-lettering or dropping outcomes, a resume landing on the wrong item, the control probe firing on the wrong steps, and a [Y] failing because provenance wasn't reconciled first. Turning each from a judgement into a counted fact makes the walk replay identically run-to-run and removes the retries that dominate its latency — without touching the part that is genuinely the model's job: the sharp, lens-specific challenge itself.
