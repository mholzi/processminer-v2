domain-architect

Develops the business-architecture layer of a process's target state — the capabilities the target process needs, the target applications that host them (with their build / buy / configure / keep verdict), and the ADRs that justify the shape — drafting each as a schema-typed element in the process JSON. It is pure downstream synthesis: it reads the documented target process and never authors upstream content.

Interactive · Y/E/R loop LAYER: business OWNS: capability · target-application · adr Invoked by: ArchitectMiner canvas buttons

a What it does

domain-architect is one of the two ArchitectMiner specialists. It facilitates a banking architect through the business-architecture layer of a process's target state and writes the result into the same process JSON via the schema-enforced tools. It owns exactly three element types: capability (what the business must be able to do, independent of how), target-application (what hosts each capability, with its build / buy / configure / keep verdict), and adr (the Architecture Decision Records that lock the shape, with alternatives and consequences written down so they survive the people who made them).

Architecture here is downstream synthesis, not fresh elicitation. Every element it drafts must be grounded in work already in the document — the target state (to-be-design), the transformation-decisions, the requirements, the gap-resolution, the dependencies, the controls, the regulation, and the as-is systems. An element grounded in nothing upstream is treated as a finding, not architecture. Because the capabilities and applications it authors are the substrate the solution-architect builds on, it usually runs first of the two.

The frontmatter it must set (the schema hard-blocks approval if missing):

  • capability — criticality (CRITICAL / HIGH / MEDIUM / LOW), reuse (NEW / REUSED); optionally owningDomain. Relations: hostedIn → target-application, realisesStep → target-state, resolvesGap → gap.
  • target-application — verdict (BUILD / BUY / CONFIGURE / KEEP); optionally vendor, owningDomain, costBand. Relation: drivenByADR → adr.
  • adr — adrStatus (DRAFT / PROPOSED / ACCEPTED / REJECTED / SUPERSEDED), owner (a role, not a named person); optionally domain. Relations: decision → transformation-decision, resolvesGap, satisfiesControl, dependsOn / supersededBy → adr.

Key constraints and behaviours:

  • Capability before application before decision. Map what the business must do first; only then what hosts it; record the ADR once the choice is real. Don't name a vendor before you know the capability it serves.
  • Build / buy is a decision, not a label. When it sets an application's verdict, it must draft the ADR that justifies it (alternatives, consequences) and wire drivenByADR. A verdict with no recorded reasoning is half an answer.
  • Recovery-safe writes. Each element is written the moment the architect confirms it via createElement — never batched in the model's head.
  • Draft, not truth-yet. Everything is authored draft with honest provenance; the architect approves later in the web app. The skill never sets approved.
  • Stays in its lane. It does not create integrations, components, NFRs or migration phases (the solution-architect's), nor any As-Is, target-state, transformation-decision, gap, requirement, control, regulation or client/innovation element — those are read-only inputs.

b The sequence

Five phases run in order; if invoked from a single section's canvas button it starts at the matching phase, otherwise it runs them all. Every draft goes through the universal Y/E/R loop with per-heading provenance.

  1. Phase 0 — OrientThe session is already scoped to one process (title, id, architect name/role). Read the upstream artifacts to ground on and the architecture already authored, so it extends rather than duplicates. Tools: expandElement() over to-be-design, transformation-decisions, requirements, gap-resolution, controls, as-is systems, plus existing capabilities / target-applications / architecture-decisions.
  2. Phase 1 — CapabilitiesDerive the business capabilities from the target-state steps and requirements. For each, set criticality and reuse, draft Description / Inputs and outputs / Boundaries, and wire realisesStep to the target-state it serves and resolvesGap to any gap it closes. Walk the whole target process so none is missed. Tools: createElement()
  3. Phase 2 — Target applicationsOpened with [A] Add one · [E] Explore · [N] None. The applications that host the capabilities — for each, its verdict (BUILD / BUY / CONFIGURE / KEEP) and reasoning, tech stack / vendor, and risks. Wire each capability's hostedIn to its application. A capability hosted nowhere, or an app hosting nothing, is a finding. Tools: createElement(), updateElement()
  4. Phase 3 — Architecture decisions[A]/[E]/[N]. Each decision as an ADR with Context / Decision / Alternatives considered / Consequences; set adrStatus and owner. Wire decision to the transformation-decision it realises, resolvesGap / satisfiesControl where they apply, and drivenByADR back from the applications a decision drives. Every non-obvious verdict earns an ADR. Tools: createElement(), updateElement()
  5. Phase 4 — Validation & closeSweep what was written for gaps: a capability tracing to no target-state step or requirement; an application with a verdict but no driving ADR; an ADR with no alternatives; a capability hosted nowhere. Surface each as a short clarifying question, then emit the fixed close with per-type drafted counts. Tools: expandElement(), updateElement()

Biggest win: push the upstream synthesis into a deterministic derive layer. Today the model must re-read the whole target state, remember every step that needs a capability, and hand-wire realisesStep / hostedIn / drivenByADR relations — the slowest and most variable part of the session. If the tool layer pre-seeds candidate capabilities from to-be-design/requirements, auto-suggests the host and ADR relations, and runs the Phase-4 graph checks itself, the architect's job collapses to confirming, culling and correcting — turning an open-ended synthesis into a fast, reproducible review while keeping every human approval intact.
