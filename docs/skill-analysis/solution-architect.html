solution-architect

Develops the technical layer of a process's target architecture — the integrations between target applications, the components inside them, the non-functional requirements (NFRs) that constrain them, and the migration phases that sequence the cutover — drafting each as a schema-typed element in the process JSON. It is pure downstream synthesis built on the domain layer and never authors upstream content.

Interactive · Y/E/R loop LAYER: technical OWNS: target-integration · component · nfr · migration-phase Invoked by: ArchitectMiner canvas buttons

a What it does

solution-architect is the second of the two ArchitectMiner specialists. It facilitates a banking architect through the technical layer of a process's target architecture and writes the result into the same process JSON via the schema-enforced tools. It owns four element types: target-integration (an interface between two target applications and what flows), component (a sub-element of a target application — a service, gateway, worker or store), nfr (a non-functional requirement with a measurable target), and migration-phase (a staged step from the as-is estate to the target).

Architecture here is downstream synthesis. The capabilities and target applications authored by the domain-architect are this skill's substrate — it connects them, decomposes them, constrains them and sequences them — so the domain architect usually runs first. Every element must be grounded in work already in the document: the capabilities and target-applications (domain layer), the requirements, controls, regulation, dependencies, and the as-is systems and integrations. An element grounded in nothing upstream is treated as a finding, not architecture.

The frontmatter it must set (the schema hard-blocks approval if missing):

  • target-integration — pattern (SYNC / ASYNC / EVENT / BATCH), direction (ONE-WAY / TWO-WAY / PUB-SUB); optionally contract, volume. Relations: from / to → target-application, realises → capability, drivenByADR → adr.
  • component — tech (the framework/runtime); inApp relation is required; optionally dataStore, hosting, scaling. Relations: dependsOn → component, realisesCapability → capability.
  • nfr — category (PERFORMANCE / AVAILABILITY / SECURITY / COMPLIANCE / SCALABILITY / …), target (a concrete, testable value); optionally owner. Relations: appliesTo → capability / component / target-application / target-integration, satisfiesControl, regulatedBy, drivenByADR.
  • migration-phase — phaseStatus (PLANNED / IN-FLIGHT / DONE), startQuarter, endQuarter; optionally owner. Relations: delivers → capability / component / target-application, dependsOn → migration-phase, resolvesGap → gap.

Key constraints and behaviours:

  • Connect, decompose, constrain, sequence — in that order. Integrations between the applications, then the components inside them, then the NFRs that constrain them, then the phases that deliver them.
  • An NFR without a measurable target is a wish. Every NFR carries a concrete, testable target ("p95 < 1.2s", "RTO ≤ 4h", "99.95% monthly") and a Measurement and Verification block. "Fast" and "secure" are prompts for a number.
  • A manual hand-off is a missing integration. When the design implies two applications exchanging data, the interface is captured with its pattern and direction — a re-key must not hide as "they'll just share the database".
  • Recovery-safe writes & draft-only. Each element is written on confirmation via createElement, authored draft with honest provenance; the architect approves later in the app. The skill never sets approved.
  • Stays in its lane. It does not create capabilities, target applications or ADRs (the domain-architect's — they are inputs), nor any As-Is, target-state, transformation-decision, gap, requirement, control, regulation or client/innovation element.

b The sequence

Six phases run in order; if invoked from a single section's canvas button it starts at the matching phase, otherwise it runs them all. Every draft goes through the universal Y/E/R loop with per-heading provenance.

  1. Phase 0 — OrientThe session is already scoped to one process. Read the domain layer to build on plus the upstream constraints, and the technical elements already authored so it extends rather than duplicates. Tools: expandElement() over capabilities, target-applications, requirements, controls, regulation, as-is systems / integrations, plus existing target-integrations / components / nfrs / migration-phases.
  2. Phase 1 — IntegrationsThe interfaces between target applications. For each: from and to applications, pattern (SYNC / ASYNC / EVENT / BATCH), direction, the contract, and the failure mode. Wire realises to the capability the interface serves. An integration naming fewer than two applications, or a data hand-off with no integration, is a finding. Tools: createElement()
  3. Phase 2 — ComponentsOpened with [A] Add one · [E] Explore · [N] None. Decompose each target application into components (services, gateways, workers, stores). For each: its inApp application (required), its tech, responsibility and technical detail, and dependsOn / realisesCapability where they apply. Tools: createElement()
  4. Phase 3 — NFRs[A]/[E]/[N]. Derive non-functional requirements from the requirements, controls and regulation. For each set category and a measurable target, draft Definition / Measurement / Verification, and wire appliesTo, satisfiesControl and regulatedBy where they apply. Tools: createElement(), updateElement()
  5. Phase 4 — Migration phases[A]/[E]/[N]. The sequence from the as-is estate to the target. For each phase: phaseStatus, startQuarter, endQuarter, the Scope, Acceptance criteria and Risks, and delivers / dependsOn / resolvesGap. Together the phases should deliver every capability and application without a big-bang cutover. Tools: createElement()
  6. Phase 5 — Validation & closeSweep what was written: an integration naming fewer than two applications; a component with no inApp; an NFR with no measurable target; a capability or application delivered by no migration phase. Surface each as a short clarifying question, then emit the fixed close with per-type drafted counts. Tools: expandElement(), updateElement()

Biggest win: derive the technical graph from the layers below it. The two slowest, most variable jobs here are reconstructing which applications must talk to each other (integrations) and proving the migration delivers everything without a big bang — both are graph problems the tool layer can solve over the as-is integrations, the domain host map, and the dependsOn edges. If integrations are seeded from the systems graph, inApp is auto-suggested, NFR targets are validated numeric at draft time, and a coverage solver flags undelivered capabilities live, the architect's role collapses to confirming patterns, numbers and sequence — a fast, reproducible technical review with every human approval still intact.
