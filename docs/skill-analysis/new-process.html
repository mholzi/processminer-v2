new-process

Scaffolds a brand-new banking process as a single strongly-typed JSON document with an empty overview — name, confirm, create. It captures no content; specialists fill it in later.

Invoked by: process switcher "new process" action, or qer-session Interactive (one short Y/E/R loop) WRITES_TO: wiki/processes/<slug>.json (root meta + empty content) OWNS: no element types

a What it does

new-process is the first skill in the documentation flow (new-process → document-ingest → triage → foundational-run). Its single job is to create the durable container — the process JSON document at wiki/processes/<slug>.json — carrying only the process identity (meta: id, slug, title, abbreviation) and an empty overview (content). It deliberately captures no content: no steps, no overview prose, no elements. When it finishes, the process appears in the app empty and ready for qer-session or a specialist to author into it.

The skill's only judgement item is a one-line description. The slug, the <PROC> abbreviation, and the slug-conflict check are derived deterministically — the skill is told to use exactly what the derivation returns and never to invent its own. The abbreviation has a hard constraint: 2–6 uppercase letters, no digits or symbols (e.g. FRD2 is rejected; use FRDT).

Key constraints and behaviours:

  • Never hand-write the file. Creation goes through the scaffoldProcess({ slug, PROC, title, description }) tool, which builds the document deterministically, validates slug + abbreviation, and refuses to overwrite an existing process.
  • Slug-conflict guard. If the derivation reports slugTaken: true, the skill must not scaffold — it tells the user the name collides and asks for a different one.
  • Fixed wording. The confirmation bullets and the closing message are verbatim — bulleted list (never a table), each label bold, and a canonical success closing that says nothing about JSON or file paths.
  • Speed-first by design. The skill's own mandate is "keep this fast: name → confirm → document created. Nothing more."

b The sequence

Four prescribed steps; the only loop is the description Y/E/R.

  1. Step 1 — NameAsk the user for the process name. That is the only thing the skill asks for.
  2. Step 2 — Derive, draft, confirmDerive the deterministic slug, the <PROC> abbreviation, and the slugTaken flag for the name. If slugTaken is true, stop and return to Step 1 with a new name. Otherwise draft a one-line description and present all three (description / slug / abbreviation) as bold-labelled bullets.
  3. Y/E/R on the descriptionOffer exactly three choices: [Y] accept → Step 3; [E] apply corrections, re-show, ask again; [R] redraft from scratch with one or two sharper questions, re-present. Loop until accepted. Create nothing until confirmed.
  4. Step 3 — ScaffoldOn confirmation, create the document. Tools: scaffoldProcess(). Do not relay the tool's printed output; on error, relay the error and stop.
  5. Step 4 — DoneOn success, emit only the canonical closing (process created, app switched, prompt to upload a document or run a documentation session). Say nothing about the JSON document or paths.
