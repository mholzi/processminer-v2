conflict-resolution

Walks the SME through the conflicts a re-ingest flagged — where a document contradicted content the wiki already held — one at a time, document version versus wiki version, applies their decision, and clears the resolved conflicts.

Invoked by: free chat / triage when a process has ingest conflicts Interactive (per-conflict D/W/E decision) WRITES_TO: conflicted element fields + clears ingest.conflicts OWNS: no element types

a What it does

conflict-resolution is the human-in-the-loop counterpart to document-ingest. When a re-ingested document contradicts content the wiki already holds, the ingest never overwrites — it records each collision in the process JSON's ingest.conflicts array and leaves the wiki element untouched. This skill is how those parked conflicts get decided.

It is strictly interactive: the SME is present and decides every conflict; the skill never resolves one itself and never invents a resolution. Each conflict carries the element id, the field (a block heading or a frontmatter field), documentSays and wikiSays. The SME chooses per conflict: take the document, keep the wiki, or edit a merged value.

Key behaviours and constraints:

  • Surgical writes. A decision changes only the conflicted block or field via updateElement({ id, patch }); the rest of the element stays byte-identical, so nothing is re-typed.
  • One at a time, never batched. Each decision is applied before moving to the next conflict, and after every [D]/[E] write the skill runs a conformance check and fixes any flag.
  • Empty-state short circuit. If conflicts is empty, the skill replies with one exact line and stops.
  • Draft, not approved. Everything written stays status: draft; the SME approves in the app. Edits reset that element's approval/provenance per the shared contract.
  • Verbatim wording. The per-conflict prompt and the final report template are reproduced exactly, never written from memory.

b The sequence

Read the parked conflicts, decide each with the SME, clear, report.

  1. Step 1 — Read the conflictsRead the ingest.conflicts array in wiki/processes/<slug>.json. If empty, reply with the exact "No conflicts to resolve" line and stop. Otherwise tell the SME how many there are and begin.
  2. Step 2 — Resolve each conflict (D/W/E)In order, present each conflict plainly (element id · field, document-says vs wiki-says) with three choices. [W] keep wiki — nothing written. [D] take document — apply via updateElement({ id, patch }). [E] work out a merged value with the SME, then apply via updateElement({ id, patch }).
  3. Conformance after every writeAfter each [D] or [E] write, Tools: checkConformance({ elementId }) — fix any flag before moving to the next conflict. Never batch decisions.
  4. Step 3 — Clear the resolved conflictsWhen every conflict has been decided, Tools: clearConflicts() — empties the ingest.conflicts array so the triage screen no longer flags them.
  5. Step 4 — ReportEmit the canonical report with the counts (document version taken / wiki version kept / merged), reproducing every other character exactly.

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

Biggest win: add a deterministic takeDocumentVersion apply path (with conformance folded in) and make conflict clearing incremental. The [D] case — "the document is right" — is the most common decision, and today it routes through the model re-authoring a patch from documentSays, which can paraphrase the captured value and forces a separate conformance round. Copying the already-verbatim document value directly, checking it in the same call, and dropping that conflict from the array on success makes the dominant path exact, single-call, and resumable — turning a serial judgement loop into a tight confirm-and-apply rhythm.
