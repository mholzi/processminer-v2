source-target

Not a web search — an internal consolidation. a What it does

source-target is the internal-consolidation analogue of source-innovation: where the other three sourcing skills pull from the web, this one synthesises the target perspective entirely from the wiki the five perspective specialists have already built. There is no web search and no external source — every block it writes is source: proposed with empty evidence, the honest default for AI synthesis the SME has not confirmed.

It produces the heart of the consolidation — target-state (in to-be-design), transformation-decision and gap — plus three section seeds so the Target Process area is not left structurally empty: requirement, process-dependency and assumption, each derived mechanically from the consolidation. The deep forward-synthesis with the SME is the transformation-agent's job; this is only the fast first stub it starts from.

Key constraints and behaviours:

  • Non-interactive. No SME, no Y/E/R, no approval — all status: draft, confidence: low.
  • Provenance = proposed, not web. This is the key difference from the three web-sourcing skills: there is no URL to cite because nothing came from the web. Every heading is proposed / evidence: "", which blocks approval until the transformation-agent walks it through the SME and flips confirmed headings to elicited.
  • Consolidate, never invent. It reads As-Is, risk, CX, innovation and IT elements and only synthesises what they document — it never creates or edits those elements, and never invents a problem, idea or capability the wiki does not hold. A thin process honestly yields a thin stub.
  • Coverage discipline. Every open problem across all perspectives should be resolved by a decision or surfaced as a gap; innovation ideas are the raw material of the target states.
  • Single cross-referenced batch. The whole Target Process is one createElements batch — decisions, gaps, requirements reference target-states by "@<tempKey>", and the writer resolves those only within one batch.

b The sequence

Six prescribed steps (incl. a section-seed step), ending in one batched write.

  1. Step 1 — Read the whole processRead the overview, then every perspective: As-Is (process-step, pain-point, process-gap, role, metric, exception), risk (control, regulation, compliance-gap, audit-finding), CX (friction-point, cx-touchpoint, moment, cx-benchmark), innovation (market-trend, competitor moves, innovation-idea, innovation-risk), IT (system, integration), and any existing target elements so it extends rather than duplicates. Tools: expandElement()
  2. Step 2 — Draft the target statesCluster the innovation-ideas by the part of the process they reshape; each genuine cluster is one target-state theme. Capture replaces (the As-Is step ids it touches), give each a tempKey, hold for the batch.
  3. Step 3 — Draft the transformation decisionsDerive a transformation-decision per target and per open problem; capture resolves (the As-Is problems it closes) and realises (the target-states by "@<tempKey>"). Walk every open problem so at least one decision resolves it. Give each a tempKey; hold.
  4. Step 4 — Draft the gapsAny open problem no decision resolves, any target no decision realises, and any innovation-risk the transformation must manage becomes a gap, linked to the target-state it serves by "@<tempKey>".
  5. Step 4.5 — Section seedsDerive the seeds mechanically: a requirement per transformation-decision (linked by temp key), a process-dependency per external feeder/consumer named in system/integration (at least two), and at least two assumption elements — all confidence: low, every block proposed.
  6. Step 5 — Write the batch & reportAssemble one manifest of every target-state, decision, gap, requirement, dependency and assumption (each omitting id, carrying its tempKey, links as "@<tempKey>"), write it in a single batch and run conformance, then emit the canonical report from the returned counts. Tools: createElements(), checkConformance()

Biggest win: turn the consolidation into deterministic rules — one gap per unresolved As-Is/To-Be delta, one requirement per decision, relations derived from the ideas' existing links — rather than free synthesis. Because source-target has no web variance (it only reads the wiki), its non-determinism comes entirely from how the model clusters ideas and judges coverage. Pushing those into rule-driven tool computation makes the first stub a near-pure function of the documented work, while a one-call full-process read removes the serial expand chain that dominates its (already non-web) latency.
