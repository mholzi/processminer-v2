add-entry

Drafts one new element into a single wiki section — read the section's context, ask the SME what to add, research it in the wiki and (where useful) on the web, then refine via Y/E/R and write it on approval. The interactive, single-entry counterpart to the bulk source-* skills.

Interactive (Y/E/R loop) WRITES_TO: one element via createElement() (status: draft) Invoked by: section "add entry" action / free chat SCOPE: one element

a What it does

add-entry appends a single AI-drafted element to the one <section> of the one process <slug> it was invoked for. It is the manual, surgical complement to the autonomous sourcing skills (source-innovation, source-cx): instead of bulk-filling a section, the SME describes a single thing they want — a market trend, an innovation idea, a competitor move, a CX benchmark, a control, or any other element type the section holds — and the skill drafts exactly that one entry.

The skill does the judgement (ask, research, draft, refine) and the schema-enforced createElement() tool does the mechanical write. Every entry is grounded: it must connect to what the wiki already holds (related pain-points, steps, trends), and external facts must cite a real source — either web research or the SME. Nothing is invented, and nothing is written until the SME accepts the draft in the Y/E/R loop.

Everything it writes is status: draft. The skill never sets approved or the relevance triage itself — the SME does that later on the element card. To add to a different section, the SME runs the skill again from there: one element per run.

b The sequence

Read the section, ask, research, draft, refine, write — five steps that bracket one Y/E/R loop.

  1. Read the section's contextRead the schema for the section's element type(s) — every ## block, frontmatter field and relation — so the draft is as complete as a dedicated sourcing skill's. List existing elements so the new one fits and does not duplicate, and read the process overview for domain context. Tools: expandElement({ type }) · expandElement({ type, id })
  2. Ask what to addAsk the SME, substituting the section label: "What would you like to add to {section label}?" Wait for the answer — never draft from a blank guess.
  3. Research, then draftAlways read the related wiki elements (an idea's pain-points, a control's steps); use web search for external signals (trends, competitor moves, benchmarks), citing the real source. Then draft every block, every frontmatter field and relation the type lists (incl. sourceUrl/asOf for web types), an honest confidence and a source. Tools: web search
  4. Refine with the SME — Y / E / RPresent the draft with exactly three choices: [Y] accept, [E] edit-and-re-show, [R] rewrite with sharper research. Loop until [Y]; always offer all three.
  5. Write itOn [Y], assemble the element object (no id — the backend assigns it) and write it as status: draft, then conformance-check and fix any flag. Confirm with the exact added-line. Tools: createElement({ type, element }) · checkConformance({ slug })

Biggest win: a scaffoldElement-style template pre-fill combined with a structured, schema-validated draft object. Together they turn the riskiest moment — assembling a complete, conformant element by hand at write time — into a fill-in-the-blanks exercise that is validated before the SME ever sees it, eliminating the post-write conformance flag and the extra [E] rounds it triggers.
