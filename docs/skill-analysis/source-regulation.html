source-regulation

Autonomously scans the web for the financial-services regulation, supervisory rules and guidance that govern a process, and drafts one regulation element per material obligation — every block sourced from a real URL, no SME present.

Non-interactive · button-invoked SOURCE: web · provenance=web FILLS: Regulation Triaged, not approved

a What it does

source-regulation is the Risk & Compliance counterpart of source-cx and source-innovation. Invoked with a process <slug> by a UI button or another skill, it reads the process, web-searches for the regulation that bears on it, and writes one regulation draft element per material obligation. The Control & Compliance Specialist later refines each one with the SME and maps it to the controls that satisfy it.

It produces exactly one element type — regulation, in the regulation section. It deliberately does not produce controls, control gaps or audit findings: those are drawn from the SME and the process itself, not the web.

Key constraints and behaviours:

  • Non-interactive. No SME is present — it asks nothing, runs no Y/E/R loop, never sets approved. Everything is written status: draft; the SME triages relevance and approves later in the app.
  • Jurisdiction-scoped. It uses the process overview's jurisdiction as the scope hint — the home-jurisdiction regulation plus the international standards (e.g. Basel) that flow into it.
  • Real sources only. It names real regulations and supervisory bodies (BaFin, ECB, EBA) and cites real URLs — never invents a regulation or a citation. If web search is unavailable, it writes only what it can solidly support and says so.
  • Provenance = web. Every block carries source: web with evidence = URL + verbatim snippet + fetch date. A web heading blocks approval until a specialist confirms it with the SME, flipping it to elicited.
  • Cross-links onto controls. Where a regulation is clearly satisfied by an existing control, it patches that control's regulatedBy — a regulation has no controls field of its own; the control list is the derived reverse.

b The sequence

Three prescribed steps; writes are incremental, not one end-of-run batch.

  1. Step 1 — Read the processRead the overview (domain, industry, jurisdiction, scope), the documented As-Is (process-steps, roles, exceptions) to know what activity is being regulated, the existing controls so a regulation can be mapped to the control that satisfies it, and any existing regulation elements so it extends rather than duplicates. Tools: expandElement()
  2. Step 2 — Scan the regulatory landscapeWeb-search across the domains that bear on the process — prudential, financial crime, conduct, operational risk, data & reporting — for named regulations and their supervisory body. Draft one regulation per material obligation (blocks What it requires / Why it applies / How it is met; frontmatter domain, source, sourceUrl) and write incrementally, keeping a running per-type total. Where a regulation maps to a control, patch that control's regulatedBy. Tools: createElements(), updateElement(), checkConformance()
  3. Step 3 — ReportRead the regulation count from the running total (summed across every createElements call, never recounted from memory) and emit the canonical report verbatim — count drafted, sources used, and the line that all are status: draft for review or refinement by the control & compliance specialist. If web search was unavailable, add the one fixed line saying so.

Biggest win: fan the regulatory domains out to parallel sub-agents driven by authored, jurisdiction-keyed query templates. The skill is almost entirely IO-bound on web search, and its five domains are independent — running them serially with improvised queries is the dominant cost and the dominant source of run-to-run variance. Parallel fixed-query sub-agents plus a deterministic dedup/cap collapse both the latency and the non-determinism at once.
