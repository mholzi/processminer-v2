process-specialist

A forensic As-Is interviewer: it walks a banking SME through how a process actually runs today and captures the mechanics as strongly-typed draft elements — steps, exceptions, pain points, gaps, roles and metrics.

Interactive · Y/E/R loop LENS: operational mechanics — who does what, in what order, where it breaks OWNS: process-step · exception · pain-point · process-gap · role · metric Invoked by: free chat / Deep Dive / qer-session

a What it does

The process specialist documents the As-Is process — the spine of the whole wiki, the elements every other perspective hangs relations off. It is forensic and methodical: it examines the process piece by piece, and every question earns its place. The discipline is you draft, the SME validates — the SME is never asked to write prose; the specialist listens, drafts the schema-shaped element itself, and lets the SME correct it.

It owns six element types plus the process overview, all in the As-Is Process area:

  • process-step — a stage of the process: what happens, inputs, outputs, the why-it-matters (non-optional — a step nobody can justify is a finding), the owning role, conditions/SLAs, and outgoing transitions.
  • exception — a deviation from the standard flow, linked back into the affecting step as an exception-kind transition.
  • pain-point — staff/process friction in running the process (re-keying, chasing, waiting). Client-facing pain is the Client Journey specialist's.
  • process-gap — something missing in how the process is run or measured (no owner, no SLA tracking). A thing the SME says is "not measured" is recorded here, not as a metric.
  • role — a participant, its responsibility, and its full RACI across the steps it touches.
  • metric — how the process is measured today (cycle time, volume, error rates).

Key disciplines baked into the skill: recovery-safe writes (each element is written the instant the SME confirms it — never batched in the model's head, so an interrupted session loses nothing), structure and traceability (a process is a graph — exceptions and pain points name the step IDs they affect; roles name the steps they touch), and a hard stay-in-your-lane rule: it never authors controls, systems, CX or target-state elements — it just flags them for the owning specialist and steers back.

b The sequence

The shared Brainstorm → Author → Verify pattern, made concrete: Brainstorm is the technique-led, narrative-first elicitation; Author is a createElement/updateElement write the moment the SME says [Y]; Verify is the per-element Y/E/R loop plus a closing gap-analysis pass. The SME picks the process and decides when each phase — and the session — is done. Phases 0, 1 and 8 are skipped under orchestrated mode (qer-session owns them).

  1. Phase 0 — SetupTake the SME name/role from the invocation as the provenance source (do not re-ask). Identify the process: review the Document Map of an existing one so you extend not duplicate, or scaffold a new one. Tools: expandElement()
  2. Phase 1 — OverviewElicit purpose, trigger, frequency, volume, in/out of scope. Draft, run Y/E/R, and on [Y] write the root content (title, purpose, trigger, frequency, scopeIn, scopeOut). Tools: updateElement()
  3. Phase 2 — Process stepsThe spine (aim 5–15 steps). Three ways in: walk it start-to-finish, draft-from-typical-and-correct, or name-phases-first. Capture what happens, in/out, why-it-matters, owning role, conditions/SLAs, and the transitions list (targetId|kind|when). Tools: createElement()
  4. Phase 3 — Exceptions[A] Add · [E] Explore · [N] None. Narrative-first: let the SME talk; you extract the fields. Write the exception, then patch the affecting step with an exception-kind transition. Tools: createElement() · updateElement()
  5. Phase 4 — Pain points[A]/[E]/[N]. Staff/process friction in running it. Capture severity, impact, root cause; link the affected step IDs. Client-facing friction is noted for the Client Journey specialist. Tools: createElement()
  6. Phase 5 — Process gaps[A]/[E]/[N]. Things missing in how the process is run or measured. Tools: createElement()
  7. Phase 6 — Roles & RACIFor each role, capture its responsibility and RACI across every step it touches — exactly one A per step. Build R/A/C/I with the SME, never guess; write as the raci relation list (e.g. ["PS-COB-001:R","PS-COB-002:A"]). Tools: createElement() · updateElement()
  8. Phase 7 — MetricsHow the process is measured today. A "not measured" answer becomes a Phase 5 gap, not a metric. Tools: createElement()
  9. Phase 8 — Validation & close-outGap-analysis sweep: steps with no owner/rationale, exceptions or pain points linked to no step, step count in 5–15, every step with ≥1 R and exactly 1 A. Surface discrepancies as questions, then print the canonical close-out verbatim. Tools: expandElement() · checkTransitions()
Throughout, the read-back rule applies: any operational detail the specialist proposed (but the SME did not literally state) is marked proposed until the SME confirms it in a read-back, at which point its heading provenance flips to elicited with the SME's quote. An element with any proposed heading cannot be approved.

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

IdeaTypeEffortWhy it helps
Structured-output authoring — have the model emit each element against the JSON Schema (the AJV "LLM output schema") natively, so createElement never bounces on a block out of word-range or a missing heading.determinismmediumEliminates the silent draft → conformance-fail → re-draft retry loop that currently burns turns; every write lands first time.
Deterministic question-bank ordering — ship a fixed, primed multiple-choice bank per phase instead of improvising questions each run.determinismlowSame SME, same process → same elicitation path; auditable and reproducible, removes run-to-run drift in what gets asked.
Batch roles into one RACI grid — present the full role×step RACI matrix once for a single Y/E/R approval instead of per-role.bothmediumRACI is a low-judgement grid once the roles are named; one approval beats N, and the matrix view makes the "exactly one A per step" rule self-checking.
Draft step list, then parallel-detail — name all phases/steps first (titles only), then draft the independent step bodies in parallel and run one batched review.speedmediumSteps are largely independent; serially drafting 5–15 one Y/E/R at a time is the slowest part of the session.
Auto-wire transitions via checkTransitions() — infer the next-step / exception transitions from the captured order and reconcile rather than hand-typing targetId|kind|when per step.determinismlowHand-formatted transition strings are an error-prone, repetitive chore; the reconcile tool already exists — lean on it instead of the model.
Pre-computed process scaffold — seed Phase 2 with a typical step skeleton for the named banking process (onboarding, payments, COB) before the first question.speedlowThe "I'll draft from what's typical and you correct it" path is the fastest; precomputing the skeleton turns authoring into correction from the first turn.
Skip read-back for document-sourced headings — when a block is lifted verbatim from an ingested source, mark it document directly instead of proposed + read-back.speedlowRead-back exists to confirm AI-inflated detail; document-grounded text already has its evidence, so the extra confirmation turn is pure overhead.
Live validation gating — run the Phase 8 gap checks (orphan exceptions, missing A, step-count band) continuously and surface them inline, not only at close.determinismmediumCatches a missing rationale or unlinked exception at the moment it happens, while the SME's context is fresh, instead of a late re-open.
One batched approval for confirmed-empty sections — when the SME says "none" to exceptions/pain/gaps, stamp the section confirmed-empty in a single deterministic action.speedlowOptional sections are common; a fixed "none" handshake is faster and more consistent than improvised wrap-up prose per section.
Biggest win: structured-output authoring driven by the JSON Schema. The process specialist writes more elements than any other specialist, and each one currently risks a conformance bounce that costs a turn and muddies provenance. Emitting schema-valid elements on the first attempt — combined with a deterministic question bank so the same process is documented the same way every run — is what turns this from an improvised interview into a reproducible, audit-grade extraction.
