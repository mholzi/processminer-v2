run-lint

The consistency checkpoint: a deterministic conformance + transitions check followed by a five-lens cross-perspective sweep of the whole process. Writes the findings via applyLint to the runtime store and re-opens every approved element a finding implicates.

Largely non-interactive WRITES_TO: lint field in data/runtime/<slug>.json via applyLint() Invoked by: "lint" CTA / free chat SCOPE: whole process

a What it does

run-lint is the council made mechanical — every perspective sweeping the whole process (SKILLS.md §9). It runs over one process <slug> in two halves. The deterministic half calls checkConformance (every element against its schema template and required frontmatter) and checkTransitions (exception wiring — orphan exceptions and dangling process-step transitions). Both return ready-shaped findings arrays that the skill must keep exactly as returned — it never authors or reshapes them.

The judgement half is a five-lens cross-perspective sweep — Process, Control & Compliance, Client Journey, Innovation, and IT Architect. Each lens reads the whole process and records discrepancies (two elements disagree, or a link is missing) and questions (the wiki is ambiguous, only the SME can resolve). Crucially the lenses route by potential relations: "a step with no control" is found by looking at steps that link to nothing.

Both halves are concatenated and handed to applyLint, which assigns finding ids, writes the lint field in the runtime store the Review panel reads, and re-opens every approved element a finding implicates (back to in-progress, stamped run-lint). It always runs — an empty pass writes an all-clear report. The skill never resolves a finding or edits an element; the SME resolves findings in the app.

b The sequence

Load, run the deterministic checks, sweep five lenses, apply, summarise — verbatim wording throughout.

  1. Load the processTake the process title from the overview. The skill orchestrates; the deterministic tools and per-lens sub-agents each read what they need, so it does not read every element itself.
  2. Deterministic checksRun conformance (element vs. template + required frontmatter) and transitions reconciliation (orphan exceptions, dangling transitions). Keep both arrays exactly as returned — a clean process returns []. Tools: checkConformance({ slug }) · checkTransitions({ slug })
  3. Cross-perspective sweep (five parallel lenses)In one message dispatch five read-only sub-agents — Process, Control & Compliance, Client Journey, Innovation, IT Architect — each sweeping the whole process through its lens and returning a JSON array of discrepancy/question findings. Wait for all five, then state one coverage line naming all five. Tools: Task (×5)
  4. Apply (deterministic)Concatenate the conformance + transitions arrays (unchanged) with the five lens arrays and apply. The tool id-stamps findings, writes the lint field in the runtime store, and re-opens implicated approvals. Always run it, even with zero findings. Tools: applyLint({ slug, findings })
  5. SummariseReport with the canonical template — discrepancy / structure / question counts and approvals re-opened — or the clean variant when there were no findings at all.
