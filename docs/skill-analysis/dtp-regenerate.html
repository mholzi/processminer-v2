DTP Enhancer

dtp-regenerate

Rewrites a bank's procedure document (DTP) from the corrected As-Is wiki and, in the same pass, critically reviews the original ingested DTP against that analysis — emitting a new versioned Markdown artifact plus one critical-review finding per material discrepancy.

Non-interactive · button-invoked WRITES_TO: runtime store + raw-sources artifact READS: As-Is wiki + original DTP Never writes the wiki JSON

a What it does

dtp-regenerate closes the loop on the As-Is. A process typically starts from an uploaded DTP — the bank's procedure document — which is then ingested and worked end to end by an analyst, so the wiki now holds a better, corrected current state than the document it came from. This skill turns that gap into two deliverables at once:

  • a regenerated DTP — the procedure rewritten from the wiki's current truth, following the original's section structure so the bank still recognises it; and
  • a critical review — the original DTP measured against the wiki, one finding per material discrepancy, each typed, severity-ranked and given a suggested disposition.

The wiki is the authority here, exactly as the source document was the authority during ingest: where the wiki and the original DTP disagree, the wiki wins, and that disagreement is a finding. It is the As-Is counterpart of area-summary — a silent generation, not a conversation.

Key constraints and behaviours:

  • Non-interactive. It reads, writes and reports — no SME questions, no Y/E/R loop, no approval gate. It is invoked by a button with a process <slug> and (usually) the original DTP filename under raw-sources/<slug>/.
  • Prefer approved content. An SME-approved element is the trusted current state and strong ground for a "the analysis found…" claim; still-draft or in-progress elements may be included but are weaker ground.
  • As-Is only. One DTP per run, from the As-Is — never the target/to-be.
  • Invents nothing. Everything in the regenerated document and every finding must trace to a wiki As-Is element or the original document; where the wiki is silent on a section the DTP covered, the section is left thin and the gap recorded as a missing/outdated finding rather than filled with plausible prose.
  • Read-only on the wiki. It never creates, edits or approves wiki elements, never resolves conflicts, and never touches the process JSON. The artifact and findings go to raw-sources/ and the runtime store — never the wiki.

b The sequence

Five steps run in order: read the old document, read the corrected wiki, regenerate, critically review, then store both.

  1. Step 1 — Read the original DTPRead the document at raw-sources/<slug>/<file> (PDF, Markdown, Word and text read directly). If no filename was given, list the folder and take the most recently ingested document, ignoring any file flagged generated in uploads.json (prior outputs of this skill). Study its section structure so the regenerated doc keeps the same shape.
  2. Step 2 — Read the corrected As-Is wikiRead the overview, then every As-Is element — process-step, role, exception, pain-point, metric, process-gap, country-variation, plus control/system where the DTP covered them. Tools: expandElement()
  3. Step 3 — Regenerate the DTPWalk the process by process-step sequence; under each step write inputs/outputs, owner, SLA, systems and applicable controls from the wiki by element title, folding in the corrections and additions the analysis produced — exceptions, pain points, gaps and controls the original omitted.
  4. Step 4 — Critically review the originalWalk the original section by section and emit one finding per material discrepancy: { kind, headline, dtpSays, wikiSays, elements, severity, rationale, suggestedDisposition }. kind is outdated / missing / contradiction / added; severity high/medium/low; disposition accepted (the DTP is wrong) or dismissed (likely the wiki is wrong).
  5. Step 5 — Store bothHand the regenerated Markdown and findings to the writer with basis: "as-is", the source filename, the markdown, the findings and coverage.dtpSections (every section walked, even findingless ones). The tool writes a versioned generated file, stamps DTPF-… ids, stores the report in the runtime store, and returns the filename + count for the one-line report. Tools: writeDtpReport()

c Enhancement ideas

Focused on making the skill more deterministic and significantly faster.

IdeaTypeEffortWhy it helps
Template-driven regeneration. Derive a deterministic Markdown skeleton from the wiki structure — one section per process-step in sequence, with owner/SLA/systems/controls slots pre-filled by the tool layer — and let the model write only the connecting prose.bothmedThe document's shape stops being a model decision: step order, ownership and control attribution come straight from the graph, so Step 3 can't drop a step or misattribute an owner, and the model's job shrinks to prose.
Deterministic section-by-section diff. Replace the holistic free comparison in Step 4 with a fixed walk: align each original-DTP section to the wiki elements that cover it, and review one aligned pair at a time.determinismmedMakes the review exhaustive and reproducible — every section is visited exactly once — instead of relying on the model to remember which sections it has already compared.
Strict findings schema, one row per discrepancy. Validate each finding against the seven-field shape (kind/severity/disposition enums included) before it is accepted into the report.determinismlowCatches a malformed or half-filled finding pre-write instead of failing in writeDtpReport, removing reject-and-retry loops and making the output auditable.
Cache the parsed As-Is wiki. Read and index every As-Is element once into an id→element map shared by both the regeneration and the review, instead of re-expanding bodies in each step.speedlowCollapses the repeated expandElement round-trips that dominate Step 2–4 wall-clock into a single read.
Parallelise the per-section review. Once sections are aligned, fan the Step 4 comparisons out across read-only sub-agents, one per section, and merge the findings.speedhighSection comparisons are independent; fanning them out turns a linear walk over a long DTP into a single concurrent pass.
Chunk large DTPs, map-reduce. Split a long original document into section chunks for the review map step, then reduce to a deduplicated finding set.speedmedKeeps each comparison inside a tight context window so a long bank procedure doesn't blow the budget or get skimmed.
Auto-derive coverage.dtpSections. Have the document parser emit the heading list directly and pass it through, rather than the model re-listing every section it walked.determinismlowGuarantees the coverage map matches the document's real headings every run, so the coverage view never silently misses a section the model forgot to name.
Deterministic element-id resolution. Wire each finding's elements array from the aligned wiki elements automatically instead of the model recalling ids by hand.determinismlowEliminates wrong or missing PS-…/CP-… ids on findings, which are what make a finding traceable back into the wiki.
Biggest win: a template-driven skeleton derived from the wiki paired with a deterministic section-by-section diff. Today both halves of the run — the regenerated prose and the critical review — depend on the model holding the whole process in its head and choosing what to compare. Letting the tool layer fix the document's structure (step order, owners, controls) and align each original section to the wiki elements that cover it makes the artifact reproducible and the review exhaustive, while the model spends its tokens on the one thing it is actually good at: judging which discrepancies matter.
