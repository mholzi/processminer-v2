comment-review

Works the open discussion comments on one element with the SME — assesses each comment's impact, incorporates the agreed changes via updateElement, flags the linked elements the change implicates, then posts a closing analyst summary into the discussion thread.

Interactive ([I]/[D]/[S] per comment) WRITES_TO: element via updateElement() + notes (resolve/summary) Invoked by: "Review with analyst" on the Discussion panel SCOPE: one element

a What it does

comment-review walks the SME through the unresolved discussion comments on a single element (process <slug>, <elementId>). For each comment it presents an impact assessment, lets the SME choose to incorporate, decline or skip it, applies the agreed edits through the normal draft flow, and closes the loop with a summary note posted into the thread.

The skill adopts the lens of the specialist that owns the element's section — read from the element's section: frontmatter and the schema's specialist mapping (process, control & compliance, client journey, innovation, transformation, or IT architect) — and signs the closing note as that analyst. If the section declares no specialist, it adopts a general process-analyst lens.

A distinctive responsibility is drift detection. Because an incorporated change can leave a linked element describing the same fact differently (a control's activity restates a step; a step's output is an exception's input), the skill reads the edited element's relations, names every implicated element by id in the summary, and — where a linked element is squarely contradicted — posts a short comment on that element so its own review picks it up. The skill edits only the one element it was invoked on; naming the others is its safe substitute for editing them.

b The sequence

Read the element and its open comments, work each one with the SME, incorporate, then close the loop.

  1. Read the element and its commentsRead the element (content, status, confidence, relations) and its notes thread; the comments to review are the unresolved ones (no resolved: true). Read the type schema and any elements the comments touch. If there are no unresolved comments, say so and stop — write nothing. Tools: expandElement({ type, id })
  2. Work each comment with the SMEOldest first, present the comment plus an impact assessment (factual correction / real gap / open question / out of scope), then offer exactly: [I] incorporate, [D] decline (capture a one-line reason), [S] skip for now. Loop until every unresolved comment has a decision; never decide for the SME.
  3. Incorporate the agreed changesFor each [I], edit the element via the patch shape that fits (block / field / list / full replace). The edit re-opens approval automatically; conformance-check and fix flags. Then read the edited element's relations and list every linked element the change now puts at odds — posting a short comment on any squarely contradicted one. Tools: updateElement({ id, patch }) · checkConformance({ slug, elementId }) · createNote()
  4. Close the loopResolve every [I] and [D] comment, then post a single summary note as the owning analyst — what was incorporated, what was declined and why, what's left open, and the implicated linked elements by id. Confirm with the exact counts line. Tools: resolveNotes({ noteIds, resolvedBy }) · createNote({ elementId, author, content, type })

Focused on making the skill more deterministic and significantly faster.

Biggest win: batching the impact assessments and decisions. Today the skill converses comment-by-comment even when comments are independent; presenting all assessments at once and collecting decisions together cuts the dominant per-comment round-trip cost, while a deterministic drift map makes the one judgement step most prone to omission — naming the linked elements a change implicates — reproducible rather than dependent on the model remembering to walk the relations.
