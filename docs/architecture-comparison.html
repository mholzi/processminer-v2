Detailed comparison

1. Process Schema Matches

Target Diagram

The orange column on the left: a single "XYZ schema" with Process Meta-Data, Roles (each with Meta Data + LLM output schema for RACI/controls/systems) and Steps (each with Meta Data + LLM output schema for step name/description).

Current Implementation

schema/process-schema.json (~2,781 lines) is the single source of truth. It defines 40 element types, their frontmatter contracts, relations, template blocks with format + word ranges, and the per-heading provenance rules.

Per-type derived files live at schema/.derived/<type>.llm.json (~50–80 lines each). The "LLM output schema" boxes in the diagram correspond to these — they are what the LLM is shown when authoring an element of a given type.

Roles and Steps in the diagram are two of the 40 actual element types; the schema generalises to all of them. The show_template.py CLI reads the derived files (not the source schema) when a skill asks for one type's contract.

2. Process JSON Different shape, intentional

Target Diagram

The blue column in the middle: a single "Process JSON" per process instance (e.g. COB-003) carrying Process Meta-Data, Roles (with Content: RACI / controls / systems), Steps, etc. One-to-many with the schema.

Current Implementation

Per process under wiki/processes/<slug>/:

  • index.md — process overview
  • Per-section folders of .md element files
  • JSON sidecars: provenance.json, raci.json, transitions.json, sections.json, lint.json, notes.json, ingest.json, review-state.json
This is the biggest structural divergence from the diagram — and it is deliberate. Karpathy's LLM-Wiki pattern (the project's foundation) keeps content as typed Markdown so per-heading provenance is human-diffable in git and every change goes through schema-validated writers. Collapsing to one JSON would force per-heading provenance into a nested object that can't be reviewed line-by-line, would lose the file-level updatedBy/updatedAt stamps git tracks for free, and would make every element edit a whole-file rewrite.

The diagram's "Process JSON" exists at runtime, not on disk: src/lib/process-view.ts's buildProcessView(doc, schema) reads all the on-disk pieces and materialises a single joined view (RACI grid, flow lanes, by-id and by-type lookups). Consumers (renderers, get_context.py, the future orchestrator) see one consolidated shape; the wiki layer stays git-native.

3. Orchestrator Matches — shipped v0.4.0

Target Diagram

Top right: a discrete "orchestrator" component that uses the Process JSON to drive the discovery workflow. Sits above the Navigation UI and feeds it.

Current Implementation

src/lib/orchestrator.ts — a pure-data state machine over ProcessView + ProcessDoc. No LLM call in the routing hot path; authoring stays LLM-driven, routing stays deterministic. Two exported functions:

  • buildOrchestratorState(view, doc) — per-process: returns ranked actions + a coarse health summary (clean, conflicts, openLintFindings, openComments, runResumable).
  • buildAttentionFeed(docs) — cross-process: splits docs into attentionRows (ranked, weighted) and cleanProcesses. Used by the WelcomeScreen ATTENTION column — verified byte-identical pre/post-migration on sepa-payments + the bank-guarantee dogfood processes.

Action vocabulary v1: resolve-ingest-conflict, resolve-lint-finding, address-comment, resume-foundational-run. Weight formula preserved verbatim from the pre-v0.4 pmAttentionForDoc: conflicts × 100 + lint × 5 + comments.

Hard principle, pinned at the top of ORCHESTRATOR-PLAN.md: never contaminate Karpathy wiki logic. The orchestrator is a read-only consumer of the wiki. It MUST NOT write to wiki/processes/<slug>/ or any sidecar, add a new sidecar of its own, stamp anything onto element frontmatter, or reinterpret a schema field outside its declared meaning. If persistence is ever needed (e.g. dismissals) it lives outside the wiki tree — .claude/, ~/.processminer/, never inside wiki/.

What landed: WelcomeScreen migrated to read buildAttentionFeed(docs).attentionRows instead of its inline pmAttentionForDoc loop. TriagePanel was inspected and deliberately not migrated — it does post-ingest-specific work (ingest receipt, confidence-spread bar, untouched-sections list) that doesn't fit the generic action vocabulary; forcing it through would have bloated the vocabulary (revision note in the plan). 13 unit tests in src/lib/orchestrator.test.ts cover weight ordering, clean-process detection, dismissed/resolved filtering, and the attention-feed split. Tagged at v0.4.0.

4. Navigation UI / Progress Menu Matches

Target Diagram

"The navigation menu is dynamically rendered from the process file." Sits between the Process JSON and the View/Render/Edit boxes.

Current Implementation

src/app/ProcessDocScreen.tsx renders areas → sections → elements hierarchically. Built per request via getProcess(slug) + buildProcessView(). Section status comes from sections.json; coverage from src/lib/coverage.ts.

The page is server-rendered with dynamic = "force-dynamic" — every request re-reads the wiki, so a Python writer mutation surfaces in the UI on the next refresh without invalidation logic.

5. Generic + Custom Renderers Matches

Target Diagram

"View / Render / Edit: section / artifact UI" branches into a Generic Renderer and a Custom Renderer (examples: RACI matrix, process flow). "All the detail views are dynamically rendered from the JSON (filtered to specific section / artifact)."

Current Implementation

  • Generic: src/components/ElementCard.tsx renders any element type's frontmatter + prose blocks
  • Custom — RACI matrix: RaciMatrix.tsx reads view.raciGrid
  • Custom — Process flow: ProcessFlow.tsx reads view.flow (lanes, stepLane, ownerOf)

Before v0.2.0-beta the join logic lived inside each renderer (a 30-line grid-pivot in RaciMatrix; a lane-assignment loop in ProcessFlow). Now both components are pure render. The diagram says "filtered down to the specific section / artifact" — that filtering is the ProcessView.byType + byId lookups.

6. Deterministic Validation Matches

Target Diagram

"Deterministic validation is done against: 1) Process JSON — e.g. dependencies, 2) Process schema — e.g. mandatory fields."

Current Implementation

  • Schema validation: scripts/wiki/check_conformance.py — required frontmatter, template heading order, format + word ranges
  • Dependency validation: src/lib/relations.ts, scripts/wiki/check_transitions.py
  • TS twin: src/lib/conformance.ts mirrors the Python conformance rules; a parity test in test_wiki_scripts.py keeps them in lock-step

Validation runs as a hard gate before every write_element.py / patch_element.py / set_approval.py call — schema-validated writers are the only mutation path into the wiki.

7. LLM Interactions Matches, with extras

Target Diagram

"Only specific artifact content + LLM output schema content is fed to LLM."

Current Implementation

scripts/wiki/get_context.py --slug --element emits a bucketed prompt:

  • STABLE — type schema (the diagram's "LLM output schema content"), process meta. Cacheable across turns.
  • VOLATILE — the focal artifact + ~30-word summaries of every directly-related element. Re-read per turn.

The diagram says "specific artifact content + LLM output schema." v0.2.0 delivers exactly that — and adds two refinements the diagram doesn't show:

  • Cache-prefix bucketing so the STABLE part reuses across consecutive turns in the same skill session
  • Related-element summaries (~30 words each) for forward + reverse links + RACI + transitions, so context scales with process size rather than exploding

All 13 specialist and worker skills now invoke get_context.py via Bash before patching existing elements (see SKILL.md WRITING-PROCEDURE-BLOCK).

What's in the code that's NOT in the diagram

Per-heading provenance contract. Every template-bearing element records the source of each ## block (elicited, document, proposed, web) plus an evidence quote where applicable. Lives in wiki/processes/<slug>/provenance.json and is enforced by scripts/wiki/check_conformance.py. See HALLUCINATION-PLAN.md. Critical for compliance use cases that the diagram doesn't address.
Approval gating. Every element flows draft → in-progress → approved. set_approval.py blocks approval while any heading is still proposed or web (unconfirmed). The web UI surfaces approval state on every element card.
Schema-validated writers as the only mutation path. The wiki is never edited directly. All writes go through write_element.py, write_elements.py, patch_element.py, set_approval.py or set_section_status.py. The diagram shows validation as a downstream gate; reality is that writers refuse to produce invalid output upstream.
Skills as Markdown. 21 skills under .claude/skills/<name>/SKILL.md each carry their own runbook + verbatim cross-skill blocks (PROVENANCE-BLOCK, WRITING-PROCEDURE-BLOCK, BATCHING-BLOCK, WEB-PROVENANCE-BLOCK) kept byte-identical via scripts/check_skill_blocks.py. Routing is by description match (the chat input fuzzy-matches the skill's frontmatter description), not by orchestrator dispatch.
Warm Claude CLI worker pool. src/lib/session-worker.ts manages a pool of pre-spawned claude processes so chat turns skip the cold start. The diagram's orchestrator implicitly assumes a single request/response shape; the real chat pipeline is streaming + worker-pooled.
Multi-module entitlements. Processminer and ArchitectMiner are co-installed in one app with per-user module entitlements (src/lib/process-access.ts). The diagram is single-module.

Net takeaway

The codebase implements the diagram's intent faithfully. After v0.4.0:

  • Six of seven concepts now match exactly — Process Schema, Orchestrator, Navigation UI, Renderers, Validation, LLM interactions.
  • One deliberate Karpathy-layer-2 divergence — the "Process JSON" lives as a runtime ProcessView, not a single on-disk file. The wiki stays typed Markdown so provenance, approval gating and git review work line-by-line; the join is reconstructed at render time and at get_context.py call time.
  • Nothing remaining pending against the original TARGET-ARCH-PLAN.md trilogy.

Beyond the diagram, the codebase carries the per-heading provenance contract, approval gating, schema-validated writers, the skill-as-Markdown runtime and a warm worker pool — none shown in the diagram but each load-bearing.
